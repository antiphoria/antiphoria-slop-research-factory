### §1. Purpose and Scope

This document specifies **every verification task** the Verifier node
performs, the **method** by which each task is executed, the
**confidence tier** each task can realistically achieve, and the
**exact tool invocation sequences** that ground the Verifier's
judgments in external evidence.

**What this document decides:**
- Which verification tasks are attempted and which are explicitly
  excluded.
- For each attempted task: whether it is performed by deterministic
  tool lookup, by LLM judgment, or by a combination.
- How the results of individual tasks compose into the five
  dimensional confidence scores defined in D-2 §8.4.
- How the five dimensional confidence scores compose into the
  single `verdict_confidence` used for routing.
- Calibration guidance for the Verifier LLM's self-reported
  confidence.
- The exact execution sequence of the Verifier node, including
  the interleaving of tool calls and LLM inference.

**What this document does NOT decide:**
- The exact text of the Verifier prompt (D-3).
- The Pydantic schemas for inputs and outputs (D-2).
- How sealing integrates with tool calls (D-5).
- Model selection (D-0).

**Key upstream constraints:**
- D-1 §10: "Citation checking must use multiple independent sources
  (Crossref + Semantic Scholar minimum). Single-source verification
  is insufficient per Attack 3."
- D-1 §10: "The Verifier system prompt must instruct the model to
  output graduated confidence, not binary verdicts."
- D-3 §4.6: Citation checks run BEFORE the Verifier LLM call
  (Option B), and results are injected into the Verifier's prompt.
- D-2 §3.4: `CitationCheckResult` has five values:
  `VERIFIED`, `METADATA_MISMATCH`, `NOT_FOUND`,
  `CLAIM_UNSUPPORTED`, `INCONCLUSIVE`.

### §2. The Verification Problem, Honestly Stated

The Aletheia papers provide the clearest empirical evidence for what
automated verification can and cannot achieve:

> "Even with its verifier mechanism, Aletheia is still more prone
> to errors than human experts." (Feng et al., 2026b, §5.3)

> "Running this prompt filtered out a P3 submission from Aletheia B
> with verdict [WRONG]. A [FIXABLE] verdict was returned on P5 and
> P7 from Aletheia A." (Feng et al., 2026a, Appendix A)

> "Aletheia A's solution [to P7] is Critically Flawed. It contains
> two arguments, both of which boil down to [a] claim... but this
> is not justified without appropriate finiteness assumptions."
> (Feng et al., 2026a, §3.2)

> "On P8, the expert assessment was not unanimous... the ambiguity
> came from subjective interpretation of whether the missing detail
> exceeded the threshold of 'minor revisions'."
> (Feng et al., 2026a, §3.2)

These observations yield three structural principles for our
Verifier:

**Principle 1: The Verifier catches some errors, not all errors.**
Even Google DeepMind's Verifier, running on the same frontier model
as the Generator (Gemini 3 Deep Think), produced false negatives
(P7-A was Critically Flawed but passed initial screening) and
debatable verdicts (P8-B split experts 5/7). Our system uses a
weaker Verifier model (Gemini Flash) on a weaker Generator model
(DeepSeek-R1). The error rate will be higher. The system's value
lies in catching *surface-level and tool-verifiable* errors, not
in providing a guarantee of correctness.

**Principle 2: Deterministic checks are strictly more reliable than LLM judgment.**
A Crossref API lookup that returns
`404 Not Found` for a DOI is a fact. An LLM's assessment that
"this logical step seems unsupported" is an opinion. The Verifier
architecture must maximize the proportion of its verdict that is
grounded in deterministic tool output, and must clearly separate
deterministic findings from probabilistic LLM assessments.
Deterministic checks are reliable about the specific property they
test; the interpretation of that result in context may still
require judgment.

**Principle 3: The Verifier must know what it doesn't know.**
Following Aletheia's self-filtering design, the Verifier must be
calibrated to output low confidence when uncertain, rather than
fabricating certainty. A `verdict_confidence` of 0.45 that
triggers a FIXABLE routing costs one revision cycle. A
`verdict_confidence` of 0.92 on a flawed draft that triggers
CORRECT approval produces sealed garbage.

### §3. The Verification Task Taxonomy

Every task the Verifier system could conceivably perform is
enumerated below, classified into one of four tiers.

#### Tier Definitions

| Tier | Name | Method | Confidence Ceiling | Meaning |
|---|---|---|---|---|
| **T1** | Deterministic | Tool API call only. No LLM judgment. | 1.0 | The result is a verifiable fact. |
| **T2** | Tool-Assisted | Tool API provides evidence; LLM interprets. | 0.8 | The result combines hard data with soft judgment. |
| **T3** | LLM-Only | No external evidence. Pure model reasoning. | 0.6 | The result is the LLM's best guess. Subject to all standard LLM failure modes. |
| **T4** | Not Attempted | Explicitly excluded from the Verifier's scope. | N/A | The task is beyond the system's reliable capability. Delegated to human review. |

**The confidence ceiling** is the maximum confidence score the
system is permitted to assign for a task at that tier, regardless
of the LLM's self-reported confidence. This is a hard architectural
cap, not a suggestion. An LLM that claims 0.95 confidence on a
T3 task is capped at 0.6 by the orchestrator.

The tier ceiling is a maximum, not a required value. Individual
tasks may specify a lower ceiling when their reliability is worse
than the rest of the tier (for example CV-5 at 0.7 and LM-4 at
0.5).

#### 3.1 Citation Verification Tasks

These tasks map to `confidence_citation_accuracy` in the
`VerifierOutput` schema (D-2 §8.4).

| Task ID | Task | Tier | Method | Confidence Ceiling | Notes |
|---|---|---|---|---|---|
| **CV-1** | DOI resolution | T1 | HTTP HEAD to `https://doi.org/{doi}`. 200 = exists, 404 = not found. | 1.0 | No LLM involved. Pure network check. |
| **CV-2** | arXiv ID resolution | T1 | HTTP HEAD to `https://arxiv.org/abs/{arxiv_id}`. | 1.0 | Same as CV-1. |
| **CV-3** | Bibliographic metadata match | T1 | Query Crossref API with DOI. Compare returned `{author, title, year}` against `CitationEntry.{claimed_authors, claimed_title, claimed_year}`. | 1.0 | Fuzzy string matching for author names (Levenshtein distance ≤ 3 per name token). Year tolerance is ±1 to accommodate online-first / proceedings mismatches. Title match via normalized Jaccard similarity ≥ 0.7. |
| **CV-4** | Paper existence by title search | T2 | Query Semantic Scholar `/paper/search?query={title}`. Evaluate all returned results and accept the best match if title similarity ≥ 0.85. | 0.8 | Fallback when no DOI/arXiv ID is provided. The search may return false positives. |
| **CV-5** | Claim-citation relevance | T2 | Retrieve abstract from Semantic Scholar, inject the abstract and specific attributed claim into the main Verifier prompt, and let the single Phase 5 Verifier inference assess whether the abstract plausibly supports the claim. | 0.7 | This is the weakest citation check. The abstract is a proxy for the full paper. A claim may be supported by a theorem in Section 4 that the abstract does not mention. |
| **CV-6** | Precise statement number plausibility | T3 | If the draft cites "Theorem 3.1 of [Reference]", the Verifier assesses whether that attribution is plausible given the abstract and surrounding context. Presence of a statement number is checked deterministically elsewhere; correctness of the number is not fully verifiable in v0.1. | 0.5 | Aletheia's Verification-and-Extraction prompt explicitly requires precise statement numbers. Our Verifier can flag their absence deterministically, but verifying that the *correct* statement number is cited requires full-paper access, which we generally do not have. |
| **CV-7** | [UNVERIFIED] tag detection | T1 | Regex scan of draft for `[UNVERIFIED]` tags. Count and list all tagged citations. | 1.0 | The Generator is instructed (D-3 §3.1) to tag uncertain citations. This is a deterministic structural check. |
| **CV-8** | Duplicate citation detection | T1 | Normalize all citation entries. Flag duplicates by DOI or by title similarity ≥ 0.9. | 1.0 | Catches the common LLM failure mode of citing the same paper under slightly different metadata. |

**Composition rule for `confidence_citation_accuracy`:**

The overall citation confidence is computed from the canonical
Phase 2 citation list plus the Phase 3 tool results. Negative
tool findings cap confidence; fully verified citations floor it
high. `INCONCLUSIVE` results degrade the score proportionally
rather than collapsing the entire dimension:
```plaintext
pseudocode:
if no citations were extracted:
    citation_confidence = min(base_confidence, 0.6)
else:
    tool_score = average({
        VERIFIED:          1.0,
        INCONCLUSIVE:      0.6,
        CLAIM_UNSUPPORTED: 0.45,
        METADATA_MISMATCH: 0.3,
        NOT_FOUND:         0.0,
    }[check.result] for check in citation_check_results)

    if all checks are VERIFIED:
        citation_confidence = max(0.9, base_confidence, tool_score)
    else:
        citation_confidence = min(base_confidence, tool_score)

    if any check.result == NOT_FOUND:
        citation_confidence = min(citation_confidence, 0.4)
    elif any check.result == METADATA_MISMATCH:
        citation_confidence = min(citation_confidence, 0.6)
```
The `base_confidence` is the Verifier LLM's self-reported
`confidence_citation_accuracy`. The composition rule ensures
that deterministic tool findings dominate the LLM's softer
self-assessment in both directions: verified citations can lift
the floor, while failed checks cap the score.

#### 3.2 Logical and Mathematical Verification Tasks

These tasks map to `confidence_logical_soundness` and
`confidence_mathematical_rigor` in the `VerifierOutput` schema.

| Task ID | Task | Tier | Method | Confidence Ceiling | Notes |
|---|---|---|---|---|---|
| **LM-1** | Structural completeness check | T1 | Regex/AST scan of draft for required sections (Introduction, Methods/Proof, Results/Conclusion, References). | 1.0 | Deterministic. Does not assess quality of content, only presence of structural elements. |
| **LM-2** | LaTeX validity check | T1 | Attempt to parse all `$...$` and `$$...$$` blocks. Flag unparseable expressions. | 1.0 | Catches broken math formatting that may mask logical errors. |
| **LM-3** | Argument chain continuity | T3 | LLM reads the draft and assesses whether each claimed result follows from the preceding steps. Flags "leaps in logic" and "unstated assumptions." | 0.6 | This is the core of what Aletheia's Verifier does — and it is the task most prone to failure. The P7-A finiteness-assumption error is a canonical example: the Verifier failed to catch that "multiplicativity of compactly supported rational Euler characteristic" requires finiteness assumptions on M. Our weaker Verifier will miss similar errors. |
| **LM-4** | Computational spot-check | T3 | LLM re-derives a specific calculation or intermediate result from the draft. Compares its own answer to the draft's claim. | 0.5 | Unreliable. The LLM may make the same computational error as the Generator, or a different one that it then incorrectly flags. Per Feng et al. (2026b, §2.3): "our results indicate that the integration of Python as a tool yielded only marginal improvements in mitigating computational hallucinations." |
| **LM-5** | Self-consistency check | T3 | LLM checks whether claims in one section of the draft contradict claims in another section. | 0.6 | More reliable than LM-3 or LM-4 because it is a comparative task (find contradictions between two specific passages) rather than an absolute assessment. |
| **LM-6** | Formal verification via Lean/Coq | T4 | **NOT ATTEMPTED.** Would require formalization of the natural-language proof into a formal proof assistant. | N/A | This is the gold standard for mathematical verification and is explicitly out of scope for v0.1. If the community formalizes a Lean integration, it would upgrade all mathematical verification tasks from T3 to T1. Listed here as a future aspiration. |

**Composition rule for `confidence_logical_soundness`:**
```plaintext
pseudocode:
Start with the LLM's self-reported score
score = verifier_output.confidence_logical_soundness
Cap at Tier ceiling: all logical tasks are T3
score = min(score, 0.6)
Bonus: if structural completeness (LM-1) passes,
the LLM had a complete draft to work with
if lm1_passed:
score = score  # no change, but log that structure was complete
Penalty: if LaTeX is broken (LM-2 fails), logical assessment
is less reliable because the Verifier may have misread
mathematical expressions
if not lm2_passed:
score = min(score, 0.3)
confidence_logical_soundness = score
```
**Composition rule for `confidence_mathematical_rigor`:**
```plaintext
pseudocode:
score = verifier_output.confidence_mathematical_rigor
score = min(score, 0.6)  # T3 ceiling
If any computational spot-check (LM-4) found a discrepancy,
rigor confidence drops hard
if lm4_found_discrepancy:
score = min(score, 0.3)
confidence_mathematical_rigor = score
```
**Why the T3 ceiling of 0.6 matters for routing:**

With `verifier_confidence_threshold` defaulting to 0.8
(D-2 §4), a purely LLM-assessed verdict can *never* achieve
CORRECT status on its own. The precise bound is stronger than the
simple "all dimensions cap at 0.6" intuition:

- `logical_soundness` has weight 0.35 and is T3-capped at 0.6.
- `mathematical_rigor` has weight 0.25 and is T3-capped at 0.6.
- Together, these two heaviest dimensions carry 0.60 of total
  weight and contribute at most `0.60 x 0.60 = 0.36`.
- Even if every remaining dimension were somehow perfect at 1.0,
  they contribute at most `0.40`.
- Therefore the absolute upper bound is `0.36 + 0.40 = 0.76`,
  which is still below the default threshold of 0.8.

This means the Verifier's CORRECT verdict will be *demoted to
FIXABLE* by the routing logic (D-3 §4.2) unless stronger non-T3
verification methods are added.

This is **intentional**. The system is architecturally designed
such that a purely LLM-verified draft cannot be approved without
either:
- Lowering the threshold (a configuration override that will
  appear in the HAI Card), or
- Tool-grounded evidence (citation checks, structural checks)
  raising the composite confidence above 0.8.

The consequence is that the factory, at default settings,
can never produce a CORRECT verdict with high confidence on
mathematical work unless formal verification tools are added.
This is the honest architectural admission that the system
produces *slop*, not *verified research*.

#### 3.3 Scope and Constraint Compliance Tasks

These tasks map to `confidence_scope_compliance` in the
`VerifierOutput` schema.

| Task ID | Task | Tier | Method | Confidence Ceiling | Notes |
|---|---|---|---|---|---|
| **SC-1** | Thesis addressal check | T3 | LLM compares the draft's central argument to `ResearchBrief.thesis`. Assesses whether the draft directly engages with the stated thesis. | 0.6 | The LLM is reasonably good at detecting gross mismatch ("the brief asked about topology but the draft discusses number theory") and poor at detecting subtle scope drift. |
| **SC-2** | Constraint compliance check | T3 | LLM reads `ResearchBrief.constraints` and checks whether the draft violates any stated constraints. | 0.6 | Example: constraint says "Use only elementary techniques" but the draft invokes Heegaard Floer homology. Per Feng et al. (2026a, §3.2, P5): "Aletheia B interpreted the 'slice filtration' in an archaic way that differs from its modern usage" — misinterpretation of problem scope is a known failure mode even at the frontier. |
| **SC-3** | Outline adherence check | T2 | If `ResearchBrief.outline` is provided, compare the draft's section headings to the outline items. Compute structural overlap. | 0.8 | Partially deterministic: heading extraction is T1, but assessing whether the content of a section matches its outline description requires T3 LLM judgment. Blended tier. |
| **SC-4** | Length compliance check | T1 | Word count of draft vs. `FactoryConfig.target_length_words`. Flag if deviation > 50%. | 1.0 | Deterministic. |
| **SC-5** | Key reference engagement check | T2 | If `ResearchBrief.key_references` is provided, search the draft's citation list for each key reference. If missing, flag. If present, LLM assesses whether the reference is substantively engaged (not merely listed). | 0.8 | Presence check is T1 (deterministic). Engagement quality assessment is T3. Blended. |

**Composition rule for `confidence_scope_compliance`:**
```plaintext
pseudocode:
scores = []
outline_structure_unknown = False
SC-1: thesis addressal (always performed)
scores.append(min(sc1_llm_score, 0.6))
SC-2: constraint compliance (only if constraints provided)
if brief.constraints:
scores.append(min(sc2_llm_score, 0.6))
SC-3: outline adherence (only if outline provided)
if brief.outline:
scores.append(min(sc3_score, 0.8))
else:
outline_structure_unknown = True
SC-4: length compliance (always performed)
if abs(word_count - target) / target > 0.5:
scores.append(0.3)  # severe deviation
elif abs(word_count - target) / target > 0.2:
scores.append(0.7)  # moderate deviation
else:
scores.append(1.0)  # within tolerance
SC-5: key reference engagement (only if key refs provided)
if brief.key_references:
for ref in brief.key_references:
if ref_found_in_draft:
engagement = min(sc5_engagement_score, 0.6)
scores.append(min(engagement + 0.2, 0.8))
else:
scores.append(0.2)  # key reference entirely absent
confidence_scope_compliance = min(scores)
if outline_structure_unknown:
confidence_scope_compliance = min(confidence_scope_compliance, 0.7)
```

#### 3.4 Novelty Plausibility Tasks

These tasks map to `confidence_novelty_plausibility` in the
`VerifierOutput` schema.

| Task ID | Task | Tier | Method | Confidence Ceiling | Notes |
|---|---|---|---|---|---|
| **NV-1** | Trivial restatement detection | T3 | LLM assesses whether the draft's central claim is a trivial restatement of a well-known result. | 0.5 | Directly motivated by the Erdős experience: "many open Erdős problems remained unresolved out of obscurity rather than difficulty" (Feng et al., 2026b, §3.3). The LLM may recognize famous results but will miss obscure ones. |
| **NV-2** | Web novelty search | T2 | Tavily search for the draft's title and central claim. If a highly similar published result is returned, flag. | 0.6 | Can catch cases where the "novel" result already exists online. Cannot catch results published in paywalled journals not indexed by Tavily. |
| **NV-3** | Full novelty assessment | T4 | **NOT ATTEMPTED.** | N/A | Per D-0 §6.1: "Novelty assessment is 'Very Low' feasibility." True novelty assessment requires knowing the complete state of a field. No LLM or search tool can provide this. Explicitly delegated to human domain experts. |

**Composition rule for `confidence_novelty_plausibility`:**
```plaintext
pseudocode:
score = verifier_output.confidence_novelty_plausibility
score = min(score, 0.5)  # T3 ceiling for the base assessment
If Tavily found a highly similar published result
if nv2_found_prior_work:
score = min(score, 0.2)
This score is ALWAYS low. By design.
confidence_novelty_plausibility = score
```
**Why this score is architecturally capped so low:**

The novelty dimension exists in the schema because the HAI Card
needs to report it and because a downstream human reviewer should
see it. But it must never influence the routing decision. A draft
that is a perfect restatement of a known result is not WRONG —
it is simply not novel. The Verifier cannot distinguish between
"this is novel and I don't recognize it" and "this is trivial and
I don't recognize it." The cap at 0.5 ensures this dimension
never inflates the overall `verdict_confidence`.

On the HAI Card, a low novelty score should be read as
"the system cannot reliably assess novelty," not "the work is
therefore unoriginal."

#### 3.5 Formatting and Presentation Tasks

These tasks do not map directly to any confidence dimension in
`VerifierOutput`. They contribute to the `critique_entries` list
as `category: "formatting"` items.

| Task ID | Task | Tier | Method | Notes |
|---|---|---|---|---|
| **FP-1** | Section structure validation | T1 | Regex check for `# Title`, `## Section`, `### Subsection`, `## References`. |
| **FP-2** | Reference section existence | T1 | Check for a `## References` section at end of draft. |
| **FP-3** | Inline citation format | T1 | Regex check: are citations in a consistent format? Are there dangling `[?]` or `[TODO]` markers? |
| **FP-4** | LaTeX rendering validity | T1 | Parse all `$...$` and `$$...$$` blocks for balanced delimiters and basic LaTeX syntax. |
| **FP-5** | `[UNVERIFIED]` tag count | T1 | Count instances of `[UNVERIFIED]` tag. Report in critique if count > 0. |

Formatting tasks are **never** the sole basis for a WRONG verdict.
They contribute MINOR severity critique entries only. A draft
with perfect logic and formatting errors receives FIXABLE, not
WRONG.

`FP-4` and `LM-2` are intentionally distinct:

- `FP-4` is a fast structural sanity check (delimiter balance,
  bracket matching, obvious malformed math).
- `LM-2` is the deeper mathematical parse used to determine
  whether broken LaTeX undermines logical assessment.

### §4. Explicitly Excluded Tasks (Tier 4)

These tasks are excluded from the automated Verifier because they
cannot be performed reliably by any combination of tools and LLMs
currently available. Each exclusion is justified.

| Task ID | Task | Why Excluded | What Would Change This |
|---|---|---|---|
| **EX-1** | Full-text citation verification | Requires downloading and reading the complete cited papers. Most papers are behind paywalls. Even if accessible, reading a 40-page paper to verify a specific claim exceeds the Verifier's reliable capability. | Open Access mandates + longer-context models with demonstrated multi-document reasoning. |
| **EX-2** | Formal proof verification (Lean/Coq) | Requires formalization of natural-language mathematics into a proof assistant's type theory. This is itself a research problem. | Automated natural-language-to-formal-language translation. Projects like LeanDojo and ProofNet are working toward this. |
| **EX-3** | Empirical reproducibility checking | Would require executing code, accessing datasets, and replicating experimental results. The factory does not execute arbitrary code. | A sandboxed code execution environment with curated datasets. Out of scope for v0.1. |
| **EX-4** | Novelty certification | Requires complete knowledge of a field's published literature, including unpublished preprints, conference talks, and folklore results. | Fundamentally impossible for any automated system. Always requires human domain expertise. |
| **EX-5** | Ethical compliance review | Requires judgment about whether the research topic, methodology, or conclusions raise ethical concerns (dual-use risks, privacy violations, etc.). | Human IRB-equivalent review. Cannot be automated. |
| **EX-6** | Peer review simulation | Attempting to simulate the full peer review process (as The AI Scientist did) produces superficial, unreliable assessments. Per the bioRxiv evaluation (research-factory-deep-research.txt): "absolutely no framework could successfully complete a full research cycle." | Fundamentally distinct from verification. Peer review assesses significance, positioning, and contribution — not just correctness. |
| **EX-7** | Image and figure verification | Publishers maintain near-total bans on AI-generated imagery (research-factory-deep-research.txt, §Publishing Landscape). The factory does not generate figures. If figures are included, they must be human-created or generated by deterministic plotting code. | Out of scope. |
| **EX-8** | Statistical methodology verification | Requires access to the data-generating process, sample sizes, model assumptions, and analysis code. The Verifier does not have this information. | Sandboxed empirical pipelines with dataset and code access, still likely requiring human review. |

### §5. The Verifier Node Execution Sequence

The Verifier node performs its tasks in a specific, fixed order.
This sequence is designed to maximize the proportion of
tool-grounded evidence available to the LLM when it makes its
judgment.

The complete sequence for a single Verifier invocation:
```plaintext
PHASE 1: DETERMINISTIC PRE-CHECKS (No LLM, No Network)
═══════════════════════════════════════════════════════
1.1  FP-1: Section structure validation
1.2  FP-2: Reference section existence
1.3  FP-3: Inline citation format check
1.4  FP-4: LaTeX rendering validity
1.5  FP-5: [UNVERIFIED] tag count
1.6  LM-1: Structural completeness check
1.7  LM-2: LaTeX validity check (deeper parse)
1.8  SC-4: Length compliance check
1.9  CV-7: [UNVERIFIED] tag detection
1.10 CV-8: Duplicate citation detection
→ Write results to workspace as pre_check_report.json
→ If LM-2 fails (broken LaTeX), flag for critique
but DO NOT abort — continue to Phase 2.
PHASE 2: CITATION EXTRACTION (Lightweight LLM Call)
═══════════════════════════════════════════════════
2.1  Send current_draft to Citation Extractor
(D-3 §6, using verifier_model at low token budget)
2.2  Parse response into list[CitationEntry]
2.3  Fallback: if LLM extraction fails, use regex
to extract entries from the ## References section
→ Write extracted citations to workspace as
citations/cycle_{NN}_extracted.json
PHASE 3: TOOL-GROUNDED CITATION CHECKS (Network I/O)
════════════════════════════════════════════════════
For each CitationEntry:
3.1  CV-1: DOI resolution (if DOI provided)
     → SEAL as TOOL_CALL step
3.2  CV-2: arXiv resolution (if arXiv ID provided)
     → SEAL as TOOL_CALL step
3.3  CV-3: Crossref metadata match (if DOI resolves)
     → SEAL as TOOL_CALL step
3.4  CV-4: Semantic Scholar title search (if no DOI)
     → SEAL as TOOL_CALL step
3.5  CV-5: Claim-citation relevance
     (if abstract retrieved from Semantic Scholar)
     → Prepare abstract/claim evidence for Phase 5
     → No separate LLM call; the single Verifier inference
       performs the relevance assessment

3.6  Compose CitationCheckEntry for this citation
3.7  Write to workspace as citations/check_{NNN}.json
3.8  Compile citation_check_summary
3.9  Write to workspace as citations/summary.json
→ If config.enable_tavily_search:
3.10 NV-2: Tavily search for title and central claim
→ SEAL as TOOL_CALL step
PHASE 4: PRE-SEAL VERIFIER INTENT
══════════════════════════════════
4.1  Render the Verifier user message (D-3 §4.2) with:
   - ResearchBrief
   - current_draft

   - citation_check_results (from Phase 3)
   - evaluation_context (cycle count, previous verdict)
4.2  Write prompt to workspace as
drafts/cycle_{NN}_verifier_prompt.md
4.3  PRE-SEAL: seal the prompt file
(step_type = PRE_VERIFIER)
PHASE 5: LLM INFERENCE (The Verifier Judgment)
══════════════════════════════════════════════
5.1  await Instructor call to Verifier model
(D-3 §4.5, using response_model=VerifierOutput)
5.2  On Pydantic validation failure after max_retries:
→ Write raw response to workspace
→ SEAL the raw response
→ Route to human_rescue_queue with error context
→ ABORT this Verifier invocation
5.3  On success: receive VerifierOutput object
PHASE 6: POST-PROCESSING (Confidence Composition)
═════════════════════════════════════════════════
6.1  Apply confidence composition rules from §3.1–§3.4:
   - Override citation confidence based on tool results
   - Cap logical/mathematical confidence at T3 ceiling

   - Cap novelty confidence at T3 ceiling
   - Compose scope compliance from sub-scores
6.2  Apply verdict demotion rule:
if verdict == CORRECT and verdict_confidence <
config.verifier_confidence_threshold:
effective_verdict = FIXABLE
# Log demotion in critique_entries as a
# category="system_override" entry
6.3  Write composed VerifierOutput to workspace as
drafts/cycle_{NN}_verifier_output.json
PHASE 7: POST-SEAL VERIFIER OUTCOME
═══════════════════════════════════
7.1  Write InferenceRecord (D-2 §9) to workspace
7.2  POST-SEAL: seal the VerifierOutput + InferenceRecord
(step_type = POST_VERIFIER)
7.3  Update FactoryState:
   - current_critique = composed_verifier_output.model_dump()
   - cycle_count += 1
   - current_extracted_citations = extracted_citations
   - citation_checks.extend(new_checks)
   - update latest_hash
7.4  Return updated state to orchestrator for routing
**Total seal events per Verifier invocation:**

| Phase | Seal Count | Step Type |
|---|---|---|
| Phase 3 (tool calls) | 2–4 per citation × number of citations | TOOL_CALL |
| Phase 3.10 (Tavily) | 0 or 1 | TOOL_CALL |
| Phase 4 (pre-seal) | 1 | PRE_VERIFIER |
| Phase 7 (post-seal) | 1 | POST_VERIFIER |
```
For a draft with 10 citations, mostly DOI-backed, the typical
arithmetic is:
- 10 DOI resolution calls
- 10 Crossref metadata calls
- 0 or a few Semantic Scholar search calls for title-only entries
- 1 Tavily call (optional)
- 1 PRE_VERIFIER seal
- 1 POST_VERIFIER seal

So a typical invocation lands around **23 seal events**, with
title-only or arXiv-heavy drafts varying modestly around that.

### §6. Tool Integration Specifications

Across all external APIs, the orchestrator should use bounded
concurrency rather than unbounded fan-out. Recommended starting
limits for v0.1:

- Crossref: up to 5 concurrent requests
- Semantic Scholar: up to 2 concurrent requests
- Tavily: 1 request at a time

If any provider returns rate-limit signals, back off and retry
rather than continuing at the same concurrency level.

#### 6.1 Crossref API

| Parameter | Value |
|---|---|
| Endpoint | `https://api.crossref.org/works/{doi}` |
| Auth | None required. Polite pool requires `mailto` in the User-Agent, e.g. `antiphoria-slop-factory/0.1 (mailto:admin@example.com)`. |
| Rate limit | 50 requests/second (polite pool) |
| Timeout | 10 seconds per request |
| Retry | 3 attempts with exponential backoff (1s, 2s, 4s) |
| Response parsing | Extract: `title`, `author[].family`, `author[].given`, `published-print.date-parts[0][0]` (year), `container-title` (journal) |

**Matching logic for CV-3 (metadata match):**

```plaintext
def match_metadata(crossref, citation) -> CitationCheckResult:
# Year: allow ±1 year to accommodate online-first and proceedings
if citation.claimed_year is not None:
if abs(crossref.year - citation.claimed_year) > 1:
return METADATA_MISMATCH
  # Title: normalized Jaccard similarity
  title_sim = jaccard(
      normalize(crossref.title),
      normalize(citation.claimed_title)
  )
  if title_sim < 0.7:
      return METADATA_MISMATCH

  # Authors: at least one surname must match
  crossref_surnames = {normalize(a.family) for a in crossref.authors}
  claimed_surnames = extract_surnames(citation.claimed_authors)
  if not crossref_surnames.intersection(claimed_surnames):
      return METADATA_MISMATCH

  # Additional guard against weak multi-author collisions:
  # if the first author does not match, require a stronger title match
  first_author_match = (
      normalize(crossref.authors[0].family) in claimed_surnames
      if crossref.authors else False
  )
  if not first_author_match and title_sim < 0.85:
      return METADATA_MISMATCH

  return VERIFIED
```

**Normalization:** lowercase, strip accents, collapse whitespace.
If Crossref returns a non-JSON response, treat it as an API error
and retry rather than attempting to parse it.

#### 6.2 Semantic Scholar API

| Parameter | Value |
|---|---|
| Endpoint (by DOI) | `https://api.semanticscholar.org/graph/v1/paper/DOI:{doi}?fields=title,authors,year,abstract,citationCount` |
| Endpoint (by title) | `https://api.semanticscholar.org/graph/v1/paper/search?query={title}&limit=3&fields=title,authors,year,abstract,citationCount` |
| Auth | API key (free tier: 100 requests/5 minutes) |
| Timeout | 15 seconds |
| Retry | 3 attempts |
| Response parsing | For search: evaluate all returned results, choose the best title match, and accept it if similarity ≥ 0.85. |

**Abstract retrieval:** If the paper is found and has an abstract,
store it in `SemanticScholarResult.abstract` for use in CV-5
(claim-citation relevance). If no abstract is available, CV-5
is skipped for this citation and the result is `INCONCLUSIVE`.

#### 6.3 Tavily Search API

| Parameter | Value |
|---|---|
| Endpoint | `https://api.tavily.com/search` |
| Auth | API key |
| Parameters | `search_depth: "advanced"`, `max_results: 5`, `include_answer: false` |
| Timeout | 20 seconds |
| Retry | 2 attempts |

**Usage contexts:**

1. **NV-2 (novelty search):** Query = draft title + first sentence
   of abstract/introduction. Examine results for highly similar
   published work.
2. **General fact-checking:** If the Verifier LLM requests a
   specific factual claim to be checked, the orchestrator can
   issue an ad-hoc Tavily query. However, in v0.1, Tavily is
   invoked only in the pre-defined NV-2 context, not dynamically
   by the LLM. Dynamic tool invocation is a Phase 2 feature.

**Security note (from D-1 §6, Attack 3):** Tavily results may
contain adversarial content designed to manipulate the Verifier.
All Tavily results are:
1. Sealed as TOOL_CALL steps (providing audit trail).
2. Injected into the Verifier prompt inside `<tool_result>` XML
   tags (per D-3 §4.2), which the model is trained to treat as
   data, not instructions.
3. Never treated as authoritative on their own. They are
   *evidence* the Verifier considers, not *verdicts*.

Before prompt injection, Tavily `content` fields are cleaned to
plaintext and truncated to a bounded length so raw HTML or prompt
injection artifacts do not bloat the context window.

### §7. Confidence Calibration Guidance

The Verifier LLM produces self-reported confidence scores.
These scores are systematically overconfident. The following
calibration table, derived from the general finding that LLMs
overestimate their own accuracy, provides adjustment factors.

This table is NOT applied automatically in v0.1. It is provided
as guidance for prompt engineering (D-3) and for future
automated calibration.

| LLM Self-Reported Confidence | Empirically Calibrated Confidence | Adjustment Factor |
|---|---|---|
| 0.95–1.0 | 0.7–0.8 | × 0.75 |
| 0.8–0.95 | 0.55–0.7 | × 0.73 |
| 0.6–0.8 | 0.4–0.55 | × 0.70 |
| 0.4–0.6 | 0.25–0.4 | × 0.67 |
| 0.0–0.4 | 0.0–0.25 | × 0.65 |

**How this interacts with the tier ceilings:**

The ordering is:
`raw score -> optional calibration -> tool-result composition -> tier cap`.
The tier ceilings (§3) are hard caps applied last. In combination:
```plaintext
pseudocode:
raw_score = llm_self_reported_confidence  # e.g., 0.9
calibrated = raw_score * calibration_factor  # e.g., 0.9 * 0.73 = 0.657
composed = combine_with_tool_results(calibrated, evidence)
capped = min(composed, tier_ceiling)
```
**Empirical validation required:** These calibration factors are
estimates. D-0 §13 Step 7 (Verifier integration test) must
include calibration benchmarks:
- Feed 20 drafts with known-correct and known-incorrect citations.
- Compare the Verifier's self-reported `confidence_citation_accuracy`
  against ground truth.
- Adjust calibration factors accordingly.

### §8. The Composite Verdict Confidence

The five dimensional confidence scores compose into a single
`verdict_confidence` that drives the routing decision.

**The composition is NOT a simple average.** Different dimensions
have different weights depending on the domain and the severity
of findings.
```plaintext
pseudocode:
def compute_verdict_confidence(v: VerifierOutput,
composed: dict, config: FactoryConfig) -> float:
"""
v: the raw VerifierOutput from the LLM
composed: the post-processed confidence scores after
tier capping and tool-result overrides
"""
  # Weight vector from FactoryConfig
  weights = {
      "logical_soundness":    config.weight_logical_soundness,
      "mathematical_rigor":   config.weight_mathematical_rigor,
      "citation_accuracy":    config.weight_citation_accuracy,
      "scope_compliance":     config.weight_scope_compliance,
      "novelty_plausibility": config.weight_novelty_plausibility,
  }
  assert abs(sum(weights.values()) - 1.0) < 1e-9

  # Weighted average
  weighted_sum = sum(
      composed[dim] * weights[dim]
      for dim in weights
  )

  # Hard floor: if any NON-NOVELTY dimension is below 0.2,
  # the composite cannot exceed 0.5
  # (a single catastrophic failure poisons the whole)
  if any(
      composed[dim] < 0.2
      for dim in weights
      if dim != "novelty_plausibility"
  ):
      weighted_sum = min(weighted_sum, 0.5)

  # Hard floor: if citations have NOT_FOUND results,
  # composite cannot exceed 0.6 regardless of other scores
  if composed["citation_accuracy"] < 0.3:
      weighted_sum = min(weighted_sum, 0.6)

  return round(weighted_sum, 3)
```
**Why novelty is weighted at only 0.05:**

Per §3.4 and the Aletheia findings, novelty assessment is the
least reliable dimension. Weighting it heavily would cause the
composite score to be dragged down by noise. The 0.05 weight
means novelty can contribute at most ±0.025 to the composite —
enough to appear in the HAI Card but too small to flip a verdict.

### §9. Verdict Routing Decision Table

Combining the `verdict` (CORRECT / FIXABLE / WRONG) with the
composite `verdict_confidence`, the routing logic makes one of
six decisions.

| Verdict | Confidence | Effective Route | Rationale |
|---|---|---|---|
| CORRECT | ≥ threshold | `finalize_manifest` | The Verifier is confident AND the verdict is CORRECT. Proceed. |
| CORRECT | < threshold | `reviser_node` (demoted to FIXABLE) | The Verifier says CORRECT but isn't confident enough. Treat as FIXABLE. Per D-3 §4.2. |
| FIXABLE | any | `reviser_node` | Identified issues. Revision warranted. |
| WRONG | any, count < max | `reviser_node` | Fatal flaw identified. The Reviser enters full-rewrite mode and treats the previous draft as failed evidence, not as scaffolding. |
| WRONG | any, count ≥ max | `human_rescue_queue` | Repeated WRONG verdicts. The system cannot self-correct. |
| any | any, total cycles ≥ max | `human_rescue_queue` | Absolute loop cap reached regardless of verdict. |

`NO_OUTPUT` remains a structural output from the Generator or
Reviser. In v0.1, the orchestrator does **not** infer `NO_OUTPUT`
from free-form Verifier prose.

### §10. What the Verifier Tells the Reviser

When the Verifier produces a FIXABLE or WRONG verdict, its output
is passed to the Reviser (D-3 §5). Not all Verifier output is
forwarded. The Reviser receives:

| VerifierOutput Field | Forwarded to Reviser? | Why / Why Not |
|---|---|---|
| `critique_summary` | ✅ Yes | Core natural-language overview of issues. |
| `critique_entries` | ✅ Yes | Itemized, actionable list of flaws. |
| `verdict` | ✅ Yes | Tells the Reviser whether to patch or rewrite. |
| `verdict_confidence` | ❌ No (default) | Per D-3 §13, Open Question: exposing confidence may cause the Reviser to argue with low-confidence critiques rather than address them. Excluded by default; can be included via config. |
| `resolution` | ✅ Yes | The remediation plan or fatal flaw detail. |
| `confidence_*` (5 dimensions) | ❌ No (default) | Same rationale as verdict_confidence. |

**What IS forwarded in addition to VerifierOutput:**

| Additional Data | Source | Why |
|---|---|---|
| `citation_check_results` (from tool checks) | `FactoryState.citation_checks` | The Reviser needs to know which citations failed DOI checks so it can remove or correct them. This is deterministic, factual data — not LLM opinion. |
| `critique_seal_hash` | `FactoryState.latest_hash` (after POST_VERIFIER seal) | Provenance linkage: the Reviser's pre-seal records which specific critique it is responding to. |

### §11. Verification Across Cycles

As the factory loops through revision cycles, the Verifier
accumulates knowledge. The following data persists across cycles
and is available to the Verifier on subsequent invocations:

| Persistent Data | Stored In | How It Helps |
|---|---|---|
| All previous `CitationCheckEntry` results | `FactoryState.citation_checks` | Citations that were `VERIFIED` in cycle 1 do not need re-checking in cycle 2. Citations that were `NOT_FOUND` should have been removed by the Reviser; if they reappear, the Verifier can immediately flag them. |
| Previous `verdict` and `critique_summary` | Injected into the Verifier prompt via `evaluation_context` block (D-3 §4.2) | The Verifier can assess whether previously identified flaws have been addressed in the revised draft. |
| `cycle_count` | `FactoryState.cycle_count` | The Verifier knows how many cycles have elapsed. On later cycles, the prompt instructs: "Pay particular attention to whether the previously identified flaws have been addressed." |

**Incremental citation checking:**

On cycle N > 1, the citation extraction step (Phase 2) produces
a new list of citations. The orchestrator diffs this against the
citations checked in previous cycles:
- Citations already `VERIFIED` with unchanged metadata: skip re-check.
- Citations newly added by the Reviser: perform full check.
- Citations previously flagged as `NOT_FOUND` that still appear:
  immediately flag with `result: NOT_FOUND` and add a
  `CritiqueEntry` with `severity: critical`,
  `category: citation_error`,
  `description: "Citation was flagged as non-existent in cycle
  {N-1} but has not been removed or corrected."`.
- Citations previously flagged as `METADATA_MISMATCH`:
  - if the metadata changed, perform the full check again
  - if the metadata did not change, escalate severity because the
    Reviser ignored factual feedback

This incremental approach reduces API calls on subsequent cycles
and catches the specific failure mode where the Reviser ignores
citation feedback.

### §12. Known Blind Spots

These are failure modes that the Verification Taxonomy
*acknowledges* but *cannot address* in v0.1. They are
documented here so that human reviewers know what to look for.

| Blind Spot | Example | Why It Escapes | Human Reviewer Action |
|---|---|---|---|
| Correct citation, wrong claim attribution | Draft cites "Smith (2020), Theorem 3.1" correctly, but Theorem 3.1 says something different from what the draft claims | CV-5 (claim-citation relevance) uses the abstract, not the full text. If the abstract doesn't mention Theorem 3.1, this escapes detection. Aletheia's own output exhibited this: "The response's citation to (Jacquet et al., 1983) should be refined to specifically (2.2) of that paper" (Feng et al., 2026a, Remark C.1). | Read the cited paper. Check the specific theorem. |
| Subtle logical fallacy | A proof step that "feels right" but contains a hidden assumption. P7-A's finiteness assumption error. | LM-3 (argument chain continuity) is T3. The LLM may make the same assumption as the Generator. | Trace the argument step by step. Check each quantifier. |
| Specification gaming | The draft "solves" the thesis by reinterpreting it in a trivially easy way. 50 of Aletheia's 63 technically correct Erdős solutions were "mathematically vacuous" due to this. | SC-1 (thesis addressal) is T3. The LLM may not recognize that the draft's interpretation differs from the intended one. | Compare the draft's interpretation to the thesis closely. Ask: "Is this what I meant?" |
| Self-consistent nonsense | A draft that is internally coherent but based on premises that are factually false. | LM-5 (self-consistency) will pass. LM-3 may pass because the reasoning chain is valid *given the false premises*. Citation checks may pass if the false premises are attributed to real papers via wrong-claim attribution (blind spot #1). | Check the foundational premises independently. |
| Think-trace anchoring of Verifier | The Verifier model generates its own `<think>` trace before producing the structured output. If the Verifier's think trace arrives at a premature conclusion ("this looks right"), the subsequent structured output may be biased toward CORRECT. | This is the same anchoring problem Aletheia's architecture was designed to address by separating Generator and Verifier (Feng et al., 2026b, §2.2). We separate them by model (DeepSeek-R1 vs. Gemini Flash), which helps but does not eliminate the Verifier's own internal anchoring. | Inspect the Verifier's critique entries. If the critique is vague but the verdict is CORRECT, be suspicious. |
| Scope creep through revision | A revised draft addresses the original thesis but adds substantial unrequested material. | SC-1 and SC-2 check whether the brief is addressed, not whether the draft stays tightly bounded to only what was requested. | Compare the final draft against the original brief and trim speculative expansions. |

### §14. Invariant Checklist for Implementation

A coding assistant implementing the Verifier node MUST verify:

| # | Invariant | How to Verify |
|---|---|---|
| 1 | Every T1 task produces a deterministic result independent of LLM state | Unit test: run the same draft through T1 tasks twice; results must be identical. |
| 2 | No T1 task ever calls the LLM | Code review: T1 functions have no `await litellm` or `instructor` calls. |
| 3 | The Verifier LLM call (Phase 5) occurs AFTER all tool calls (Phase 3) | Integration test: inspect the ordering of seal events in the chain. |
| 4 | Citation check results from Phase 3 appear in the Verifier prompt | Integration test: render the prompt and search for the citation results block. |
| 5 | Tier ceilings are applied to confidence scores AFTER the LLM call, not during | Code review: the composition functions in Phase 6 apply `min(score, ceiling)`. |
| 6 | The verdict demotion rule fires when `verdict_confidence < threshold` | Unit test: construct a `VerifierOutput` with `verdict=CORRECT`, `verdict_confidence=0.7`, `threshold=0.8`. Assert effective verdict is `FIXABLE`. |
| 7 | Each tool call is sealed as a `TOOL_CALL` step | Integration test: run the Verifier on a draft with 3 citations. Assert the chain contains the correct number of TOOL_CALL seals. |
| 8 | Incremental citation checking skips previously VERIFIED citations | Integration test: run 2 cycles. In cycle 2, assert that VERIFIED citations from cycle 1 do not generate new API calls. |
| 9 | The composite `verdict_confidence` respects the hard floor rules | Unit test: set `confidence_citation_accuracy = 0.1`. Assert composite ≤ 0.5 regardless of other dimensions. |
| 10 | A draft where all verification is LLM-only (no tool results) cannot achieve `verdict_confidence ≥ 0.8` at default settings | Mathematical proof: the two heaviest dimensions, `logical_soundness` (0.35) and `mathematical_rigor` (0.25), are T3-capped at 0.6. They therefore contribute at most `0.6 * 0.60 = 0.36`. Even if every remaining dimension were 1.0, the total would be `0.36 + 0.40 = 0.76 < 0.8`. QED. |
| 11 | Novelty plausibility never triggers the catastrophic-failure hard floor by itself | Unit test: set only `confidence_novelty_plausibility = 0.1` while every other dimension is healthy. Assert the non-novelty hard floor does not fire. |

### §15. Evolution Path

As the system matures and the community contributes, the
following upgrades would shift specific tasks to higher tiers:

| Current Task | Current Tier | Upgrade | New Tier | Dependency |
|---|---|---|---|---|
| LM-3 (argument chain) | T3 (0.6) | Integrate Lean 4 type-checker for formalized proofs | T1 (1.0) | Automated NL-to-formal translation |
| LM-4 (computational spot-check) | T3 (0.5) | Sandboxed Python/SageMath execution for numerical claims | T1 (1.0) | Code execution sandbox |
| CV-5 (claim-citation relevance) | T2 (0.7) | Full-text retrieval via Unpaywall/CORE API + long-context LLM reading | T2 (0.85) | Open Access full-text APIs |
| CV-6 (statement number) | T3 (0.5) | Full-text retrieval + targeted section extraction | T2 (0.75) | Same as above |
| NV-2 (novelty search) | T2 (0.6) | Semantic Scholar citation graph analysis: is the claim already implied by known results? | T2 (0.7) | Semantic Scholar advanced API |
| SC-2 (constraint compliance) | T3 (0.6) | Formalize constraints as checkable predicates (e.g., "no references after 2020" → deterministic date check) | T1 for formalized constraints, T3 for others | Constraint formalization grammar |

Each upgrade would be documented as a new version of this
document. The tier ceilings and composition rules would be
updated accordingly. The net effect of these upgrades would be
a gradual increase in the maximum achievable `verdict_confidence`
without lowering the `verifier_confidence_threshold` — meaning
the system would produce more CORRECT verdicts *because its
verification is actually better*, not because it has been told
to be less careful.

This is the path from slop-research-factory to research-factory.
The architecture stays the same. The tiers improve.
