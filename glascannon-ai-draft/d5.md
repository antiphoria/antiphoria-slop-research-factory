### §1. Purpose and Scope

This document specifies the exact integration surface between the
antiphoria slop-research-factory and the existing provenance seal
engine (`slop-cli`). It defines the Python wrapper API, the
precise calling sequences for every node type, the manifest
compilation procedure, the crash-recovery re-verification protocol,
and the error-handling contract.

**What this document decides:**
- The assumed CLI surface of `slop-cli` (or equivalent seal tool).
- The async Python wrapper that all nodes call.
- The exact sequence of seal events for every node type
  (Generator, Verifier, Reviser, Citation Checker, Finalize).
- The chain verification procedure, including timestamp
  plausibility checks.
- The crash-recovery protocol with mandatory re-verification.
- The manifest compilation algorithm.
- Performance budgets and failure modes.

**What this document does NOT decide:**
- The internal implementation of the seal engine itself
  (assumed to exist and work).
- Prompt text (D-3).
- Model selection (D-0).
- Deployment configuration (D-9).

**Key upstream constraints:**
- D-1 §10: "Crash recovery MUST re-verify the full chain before
  resuming."
- D-1 §10: "The seal engine must accept raw API response bytes
  as input (not just processed files) to enable future
  provider-signing integration."
- D-1 §10: "The `verify_chain` function must check timestamp
  plausibility (inter-seal intervals consistent with expected
  inference latency)."
- D-2 §7: Defines `SealPayload`, `SealedStepReceipt`,
  `ProvenanceManifest`, and workspace file layout.
- D-4 §5: The Verifier node produces approximately 22 seal
  events per invocation (tool calls + pre/post seals).

### §2. Assumed Seal Engine Interface

The provenance engine is treated as an **opaque external tool**.
This document specifies what we send to it and what we expect
back. The factory code never implements cryptographic primitives
directly.

#### 2.1 CLI Surface Assumption

The seal engine is assumed to expose three commands. The exact
binary name is configurable; we use `slop-cli` throughout this
document as a placeholder.

**Command 1: `seal`**
slop-cli seal 
--file <path_to_payload_json> 
--prev-hash <hex_string> 
--algorithm <hash_algorithm> 
[--output <path_to_receipt_json>]
| Parameter | Type | Description |
|---|---|---|
| `--file` | Path | Path to the `SealPayload` JSON file to be sealed. |
| `--prev-hash` | String | The `seal_hash` from the previous step's receipt. For the genesis seal, this is the empty string `""`. |
| `--algorithm` | String | Hash algorithm identifier. Default: `"sha256"`. |
| `--output` | Path (optional) | Path where the receipt JSON will be written. If omitted, receipt is written to stdout. |

**Output:** A JSON object conforming to `SealedStepReceipt`
(D-2 §7.2), written to the specified output path or stdout.

**Exit codes:**
- `0`: Success.
- `1`: File not found or unreadable.
- `2`: Invalid `prev-hash` format.
- `3`: Hash algorithm not supported.
- `99`: Internal error.

**Command 2: `verify`**
slop-cli verify 
--chain-dir <path_to_chain_directory> 
[--output <path_to_report_json>]
| Parameter | Type | Description |
|---|---|---|
| `--chain-dir` | Path | Path to the `chain/` directory containing all seal receipts. |
| `--output` | Path (optional) | Path for the verification report JSON. |

**Output:** A JSON verification report (see §9 for schema).

**Exit codes:**
- `0`: Chain is valid.
- `1`: Chain is broken (specific failure in report).
- `2`: Chain directory not found or empty.
- `99`: Internal error.

**Command 3: `hash`**
slop-cli hash 
--file  
--algorithm <hash_algorithm>
**Output:** The hex-encoded hash of the file contents, written
to stdout with a trailing newline.

This is a utility command used to compute content hashes for
files before constructing `SealPayload` objects. It ensures
the factory code uses the exact same hashing implementation as
the seal engine, avoiding discrepancies.

#### 2.2 What the Factory Never Does

The factory code **never**:
- Computes SHA-256 or any hash directly via Python's `hashlib`.
  All hashing goes through `slop-cli hash` to ensure
  implementation consistency.
- Modifies a file after it has been sealed.
- Reads a sealed file and re-writes it (even with identical
  content — the timestamp would differ).
- Catches and silently ignores a non-zero exit code from
  `slop-cli`.

### §3. The Async Python Wrapper

All seal operations are executed via an async wrapper class that
encapsulates subprocess calls, error handling, and state
mutation. Every node in the factory calls this wrapper; no node
calls `slop-cli` directly.

#### 3.1 Wrapper Interface
```python
class SealEngine:
"""Async wrapper around the slop-cli seal engine.
All methods are coroutines. All methods raise
SealError on non-zero exit codes.
"""

def __init__(self,
             cli_path: str = "slop-cli",
             algorithm: str = "sha256"):
    ...

async def hash_file(self, file_path: str) -> str:
    """Compute the content hash of a file.

    Returns: hex-encoded hash string.
    Raises: SealError if the file does not exist or
            the CLI returns non-zero.
    """

async def seal(self,
               payload_path: str,
               prev_hash: str,
               receipt_path: str) -> SealedStepReceipt:
    """Seal a payload file and write the receipt.

    Args:
        payload_path: Path to the SealPayload JSON file.
        prev_hash: The seal_hash from the previous receipt.
        receipt_path: Path where the receipt JSON will be
                      written.

    Returns: Parsed SealedStepReceipt dataclass.
    Raises: SealError on any failure.
    """

async def verify_chain(self,
                       chain_dir: str) -> VerificationReport:
    """Verify an entire chain directory.

    Returns: Parsed VerificationReport.
    Raises: SealError if the chain directory is missing.
    """

async def hash_bytes(self, data: bytes) -> str:
    """Compute the content hash of raw bytes.

    Writes bytes to a temporary file, hashes via CLI,
    deletes the temporary file. Used for raw API response
    sealing (D-1 §10, future provider-signing support).

    Returns: hex-encoded hash string.
    """
```
#### 3.2 `SealError` Exception
```python
class SealError(Exception):
"""Raised when a seal engine operation fails."""
exit_code: int
stderr: str
command: str         # The full CLI command that failed
step_index: int | None  # The step being sealed, if known
```
**Error handling contract:** When a `SealError` is raised:

1. The orchestrator logs the full error to
   `{workspace}/errors/seal_error_{step_index}.json`.
2. The orchestrator sets `state.status = RunStatus.FAILED`.
3. The orchestrator sets `state.last_error` to a summary string.
4. The orchestrator does NOT attempt to continue the chain.
   A broken seal is a terminal condition.
5. The workspace is preserved for forensic inspection.

#### 3.3 Subprocess Execution

The wrapper executes `slop-cli` via `asyncio.create_subprocess_exec`.
Key parameters:

- `stdout=asyncio.subprocess.PIPE`
- `stderr=asyncio.subprocess.PIPE`
- Timeout: 30 seconds per invocation. If the seal engine hangs
  (e.g., disk full, I/O deadlock), the subprocess is killed
  after 30 seconds and a `SealError` is raised.

**Why not use a Python library directly?** The seal engine is
a separate, independently auditable binary. By communicating
exclusively through the filesystem and CLI, we maintain a clean
trust boundary. The factory code cannot accidentally interfere
with the hashing implementation. An auditor can verify the seal
engine independently of the factory code. This separation is a
deliberate security architecture decision, not a convenience
choice.

### §4. The Four-Phase Node Protocol

Every cognitive node (Generator, Verifier, Reviser) follows an
identical four-phase execution protocol. This protocol is the
core mechanism that enforces the Glass Box Completeness Invariant.
```plaintext
┌─────────────────────────────────────────────────┐
│              NODE EXECUTION PROTOCOL             │
│                                                  │
│  Phase 1: PRE-SEAL                               │
│    → Write intent payload to disk                │
│    → Seal it                                     │
│    → Update state.latest_hash                    │
│    → Update state.step_index                     │
│                                                  │
│  Phase 2: INFERENCE                              │
│    → await LLM API call                          │
│    → Capture full response (think + output)      │
│    → Write raw response to disk                  │
│                                                  │
│  Phase 3: POST-SEAL                              │
│    → Write processed output to disk              │
│    → Seal it                                     │
│    → Update state.latest_hash                    │
│    → Update state.step_index                     │
│                                                  │
│  Phase 4: STATE UPDATE                           │
│    → Update state.current_draft / critique       │
│    → Append to state.messages                    │
│    → Write state.json to disk                    │
│    → Return updated state to orchestrator        │
│                                                  │
└─────────────────────────────────────────────────┘
```
**Critical ordering invariant:** Phase 1 MUST complete before
Phase 2 begins. Phase 2 MUST complete before Phase 3 begins.
There is no parallelism within a single node execution. The
four phases are strictly sequential.

**Rationale from Aletheia:** This mirrors the structure
described in Feng et al. (2026a): "The outputs of Aletheia
were filtered, without any intermediate alteration, through a
pre-determined verification and extraction prompt." The
pre-seal captures the intent *before* any LLM interaction.
The post-seal captures the result *after*. The chain proves
the temporal ordering.

### §5. Seal Sequences Per Node Type

For each node type, we specify the exact files written, the
exact seal events, and the exact `SealPayload.metadata`
contents.

#### 5.1 Genesis Seal

Executed once at run initialization, before any node runs.

**Trigger:** `FactoryState` is created and the `ResearchBrief`
has been validated.

**Files written:**
- `{workspace}/brief.json` — The serialized `ResearchBrief`.
- `{workspace}/chain/000_genesis.json` — The genesis seal.

**Sequence:**
```plaintext
pseudocode:


Step 0: GENESIS


Write brief.json to workspace.


content_hash = await seal_engine.hash_file(brief.json)


Construct SealPayload:
run_id:       state.run_id
step_index:   0
step_type:    "GENESIS"
timestamp:    now_utc_iso()
prev_hash:    ""
content_hash: content_hash
metadata: {
"config": state.config as dict,
"brief":  state.brief,
"factory_version": FACTORY_VERSION
}


Write SealPayload to chain/000_genesis.json


receipt = await seal_engine.seal(
payload_path = "chain/000_genesis.json",
prev_hash    = "",
receipt_path = "chain/000_genesis.json"
)
NOTE: The receipt OVERWRITES the payload file.
The receipt JSON includes all payload fields PLUS
the seal_hash. This is the canonical on-disk format.


state.latest_hash = receipt.seal_hash


state.step_index = 0


Write state.json
```

**Post-condition:** `state.latest_hash` contains the genesis
hash. `state.step_index == 0`.

#### 5.2 Generator Node
```plaintext
pseudocode:
**Pre-seal metadata (`PRE_GENERATOR`):**
metadata: {
"prompt_hash": str,   # hash of the prompt file
"model":       str,   # config.generator_model
"cycle":       int    # state.cycle_count + 1
}
**Post-seal metadata (`POST_GENERATOR`):**
metadata: {
"draft_hash":    str,         # hash of the output file
"think_hash":    str | null,  # hash of think trace file
"raw_response_hash": str,     # hash of complete API response
"model":         str,
"token_counts": {
"input":  int,
"output": int,
"think":  int | null
},
"cycle":         int,
"api_response_id": str | null  # from LiteLLM, for future
# provider-signing (D-1 §7)
}
```
**Files written during the Generator node:**

| Phase | File | Content |
|---|---|---|
| Phase 1 | `drafts/cycle_{NN}_generator_prompt.md` | The rendered prompt (system + user) |
| Phase 2 | `drafts/cycle_{NN}_generator_response.md` | Full API response (think + output, as received) |
| Phase 2 | `drafts/cycle_{NN}_generator_output.md` | Final output only (think tokens stripped) |
| Phase 2 | `drafts/cycle_{NN}_generator_think.md` | Think tokens only (if `capture_think_tokens` is true) |
| Phase 3 | `drafts/cycle_{NN}_generator_record.json` | `InferenceRecord` (D-2 §9) |

**Complete seal sequence for the Generator node:**
```plaintext
pseudocode:

GENERATOR NODE — Cycle C
PRE-SEAL (Phase 1):
step_index = state.step_index + 1


Render Generator prompt (D-3 §3.1 + §3.2).


Write prompt to drafts/cycle_{C}_generator_prompt.md


prompt_hash = await seal_engine.hash_file(prompt_file)


Construct SealPayload (step_type = "PRE_GENERATOR")


Write to chain/{NNN}_pre_generator.json


receipt = await seal_engine.seal(...)


state.latest_hash = receipt.seal_hash


state.step_index = step_index


INFERENCE (Phase 2):


response = await litellm.acompletion(
model = config.generator_model,
messages = [system_prompt, user_message]
)


Extract: think_tokens, final_output, token_counts,
api_response_id


Write full response to
drafts/cycle_{C}_generator_response.md


Write final output to
drafts/cycle_{C}_generator_output.md


If config.capture_think_tokens and think_tokens:
Write think tokens to
drafts/cycle_{C}_generator_think.md


NOTE: If the response starts with "NO_OUTPUT":
→ Still proceed to POST-SEAL. The NO_OUTPUT
declaration is sealed as a legitimate output.
→ state.status will be set to NO_OUTPUT in Phase 4.
POST-SEAL (Phase 3):
step_index = state.step_index + 1


draft_hash = await seal_engine.hash_file(output_file)


think_hash = await seal_engine.hash_file(think_file)
if think file exists, else null


raw_response_hash = await seal_engine.hash_file(
response_file)


Construct InferenceRecord (D-2 §9).
Write to drafts/cycle_{C}_generator_record.json


Construct SealPayload (step_type = "POST_GENERATOR")
content_hash = draft_hash
metadata includes all hashes and token counts


Write to chain/{NNN}_post_generator.json


receipt = await seal_engine.seal(...)


state.latest_hash = receipt.seal_hash


state.step_index = step_index


STATE UPDATE (Phase 4):


state.current_draft = final_output


state.current_think_trace = think_tokens


state.messages.append({
"role": "generator",
"step_index": state.step_index,
"timestamp": now_utc_iso(),
"model": config.generator_model,
"prompt_hash": prompt_hash,
"response_hash": raw_response_hash,
"token_counts": {
    "input": ..., "output": ..., "think": ...
}
})


state.updated_at = now_utc_iso()


Write state.json to workspace


Return state to orchestrator

```
**Total seal events for the Generator node: 2** (1 pre + 1 post).

#### 5.3 Verifier Node

The Verifier node is the most complex in terms of seal events
because it includes tool calls (D-4 §5, Phases 1–3) interleaved
with the LLM inference.

**Complete seal sequence for the Verifier node:**
```plaintext
pseudocode:

VERIFIER NODE — Cycle C
═══ PHASE 1: DETERMINISTIC PRE-CHECKS (D-4 §5, Phase 1) ═══
No seals. Results written to workspace as
drafts/cycle_{C}_pre_check_report.json.
These are deterministic and reproducible from the draft;
sealing them separately is unnecessary.
═══ PHASE 2: CITATION EXTRACTION (D-4 §5, Phase 2) ═══
No seal. This is a lightweight LLM call that produces
a list[CitationEntry]. The extracted citations are written
to citations/extracted.json. The cost of sealing this
intermediate step is not justified; the extracted citations
will be visible in the Verifier prompt (which IS sealed).
═══ PHASE 3: TOOL-GROUNDED CITATION CHECKS (D-4 §5, Phase 3) ═══
For each CitationEntry (index I):
TOOL_CALL SEAL — Crossref lookup (if DOI provided):
  step_index = state.step_index + 1


  1. Construct CrossrefQuery (D-2 §11.1)
  2. Execute HTTP request to Crossref API

  3. Parse CrossrefResult
  4. Write query + result to tools/crossref_{I}.json

  5. content_hash = hash of tools/crossref_{I}.json
  6. Construct SealPayload:

       step_type:   "TOOL_CALL"
       metadata: {
           "tool_name":    "crossref",
           "query":        query_summary_string,
           "response_hash": content_hash,
           "parent_step":  step_index of PRE_VERIFIER
                           (will be set retroactively;
                            see note below)
       }

  7. Write to chain/{NNN}_tool_call.json
  8. receipt = await seal_engine.seal(...)

  9. state.latest_hash = receipt.seal_hash
  10. state.step_index = step_index

TOOL_CALL SEAL — Semantic Scholar lookup:
  (Same sequence as Crossref, writing to
   tools/semantic_scholar_{I}.json)

TOOL_CALL SEAL — Tavily search (if enabled, NV-2):
  (Same sequence, writing to tools/tavily_{I}.json)
After all citation checks:
11. Compile CitationCheckEntry for each citation
12. Write to citations/check_{I}.json

13. Compile summary to citations/summary.json

(No seal for summary; it is derivable from the
 individual check files which are sealed.)
═══ PHASE 4: PRE-SEAL VERIFIER INTENT ═══
step_index = state.step_index + 1


Render Verifier prompt (D-3 §4.2) with:

brief, draft, citation_check_results, context



Write to drafts/cycle_{C}_verifier_prompt.md


prompt_hash = hash of prompt file


Construct SealPayload (step_type = "PRE_VERIFIER")
metadata: {
"prompt_hash":   prompt_hash,
"model":         config.verifier_model,
"draft_version": C,
"cycle":         C,
"num_citations_checked": int,
"tool_call_steps": [list of step_indices
for all TOOL_CALL seals above]
}


Write to chain/{NNN}_pre_verifier.json


receipt = await seal_engine.seal(...)


state.latest_hash = receipt.seal_hash


state.step_index = step_index


═══ PHASE 5: LLM INFERENCE ═══


verifier_output = await instructor_call(
model = config.verifier_model,
response_model = VerifierOutput,
messages = [system, user]
)


On validation failure after max retries:
→ Write raw response to
drafts/cycle_{C}_verifier_raw_error.md
→ Proceed to POST-SEAL with error metadata
→ Route to human_rescue_queue


Write VerifierOutput to
drafts/cycle_{C}_verifier_output.json


Write full response to
drafts/cycle_{C}_verifier_response.md


═══ PHASE 6: POST-PROCESSING ═══


Apply confidence composition rules (D-4 §3.1–3.4)


Apply verdict demotion rule (D-4 §8)


Write composed output to
drafts/cycle_{C}_verifier_output.json
(overwrite with composed version)


═══ PHASE 7: POST-SEAL VERIFIER OUTCOME ═══
step_index = state.step_index + 1


critique_hash = hash of verifier_output.json


Construct InferenceRecord
Write to drafts/cycle_{C}_verifier_record.json


Construct SealPayload (step_type = "POST_VERIFIER")
metadata: {
"critique_hash": critique_hash,
"verdict":       effective_verdict,
"confidence":    verdict_confidence,
"model":         config.verifier_model,
"token_counts":  {...},
"cycle":         C
}


Write to chain/{NNN}_post_verifier.json


receipt = await seal_engine.seal(...)


state.latest_hash = receipt.seal_hash


state.step_index = step_index


═══ STATE UPDATE ═══


state.current_critique = verifier_output.model_dump()


state.cycle_count += 1


state.citation_checks.extend(new_checks)


Update rejection_count or revision_count per verdict


state.messages.append({...})


state.updated_at = now_utc_iso()


Write state.json


Return state to orchestrator

```

**Total seal events for the Verifier node:**
- Tool calls: 2–3 per citation × number of citations + 0–1 Tavily
- Pre-seal: 1
- Post-seal: 1
- Example for 8 citations, 2 sources each, 1 Tavily search:
  `(8 × 2) + 1 + 1 + 1 = 19 seals`

**Note on `parent_step` in TOOL_CALL metadata:**
The `parent_step` field in TOOL_CALL metadata references the
step_index of the PRE_VERIFIER seal that initiated the tool
checks. Because tool calls occur BEFORE the PRE_VERIFIER seal
in the execution sequence (D-4 §5, Option B: citation checks
before Verifier), the `parent_step` cannot be known at tool-call
time. Instead:
- During tool call sealing, `parent_step` is set to `-1`
  (placeholder).
- The PRE_VERIFIER seal's metadata includes
  `tool_call_steps: [list of step indices]`, which serves
  as the authoritative forward reference.
- An auditor reconstructs the relationship by reading the
  PRE_VERIFIER metadata.

This avoids a circular dependency in the chain.

#### 5.4 Reviser Node

The Reviser follows the identical four-phase protocol as the
Generator (§5.2) with these differences:

| Difference | Generator | Reviser |
|---|---|---|
| Step types | `PRE_GENERATOR`, `POST_GENERATOR` | `PRE_REVISER`, `POST_REVISER` |
| File prefix | `cycle_{C}_generator_` | `cycle_{C}_reviser_` |
| Pre-seal metadata | `prompt_hash, model, cycle` | `prompt_hash, model, cycle, critique_hash` |
| Prompt rendering | D-3 §3.1 + §3.2 | D-3 §5.1 + §5.2 |

**Additional pre-seal metadata for the Reviser:**
metadata: {
"prompt_hash":    str,
"critique_hash":  str,   # hash of the VerifierOutput that
# triggered this revision
"critique_step":  int,   # step_index of the POST_VERIFIER
# seal that produced the critique
"model":          str,
"cycle":          int
}
The `critique_hash` and `critique_step` fields create an
explicit causal link in the chain: "this revision was
produced in response to critique X, sealed at step Y."

**Total seal events for the Reviser node: 2** (1 pre + 1 post).

#### 5.5 Human Gate Seal

When the orchestrator routes to `human_rescue_queue`, a
`HUMAN_GATE` seal is produced to record the routing decision.

When the human responds, a second `HUMAN_GATE` seal records
their decision.
```plaintext
pseudocode:

**Routing seal:**
step_type: "HUMAN_GATE"
metadata: {
"direction":     "to_human",
"reason":        "max_rejections_exceeded" |
"max_revisions_exceeded" |
"max_total_cycles_exceeded" |
"verifier_parse_failure",
"latest_verdict": str | null,
"cycle":          int
}
**Response seal:**
step_type: "HUMAN_GATE"
metadata: {
"direction":      "from_human",
"decision":       str,    # HumanRescueResponse.decision
"reviewer_id":    str | null,
"comment":        str | null,
"modified_draft": bool    # whether the human provided
# a modified draft file
}
```
If the human provides a modified draft, the modified draft
file is written to `drafts/cycle_{C}_human_edit.md` and its
content hash is included in the seal metadata.

#### 5.6 Manifest Seal

The final seal in the chain. Produced by the FINALIZE node
after all cycles complete.

```plaintext
pseudocode:

step_type: "MANIFEST"
metadata: {
"total_steps":     int,
"final_verdict":   str,   # Verdict.value
"final_confidence": float,
"output_hash":     str,   # hash of final paper.md
"hai_card_hash":   str,   # hash of hai_card.md
"total_cycles":    int,
"total_duration_seconds": float
}
```
The manifest seal's `content_hash` covers the compiled
`ProvenanceManifest` JSON itself (see §8).

### §6. Raw API Response Sealing

Per D-1 §10: "The seal engine must accept raw API response
bytes as input (not just processed files) to enable future
provider-signing integration."

#### 6.1 The Problem

The Generator and Reviser receive API responses that contain
both `<think>` tokens and final output in a single response
object. The factory currently writes these as separate files
(think trace, final output, full response). An adversary who
controls the machine (D-1 §6, Non-Defense 1) could substitute
the final output while preserving the think trace.

#### 6.2 The Mitigation

In addition to the per-file hashes in the POST-SEAL metadata,
every inference call produces a `raw_response_hash` computed
over the **exact byte sequence** returned by the LiteLLM
client, before any parsing or splitting.
pseudocode:
Immediately after the API returns
raw_bytes = response.to_json().encode("utf-8")
raw_response_hash = await seal_engine.hash_bytes(raw_bytes)
Write raw bytes to disk
with open(response_file, "wb") as f:
f.write(raw_bytes)
THEN parse into think_trace and final_output
think_trace, final_output = parse_response(response)
The `raw_response_hash` is recorded in the POST-SEAL metadata.
An auditor can verify that:
1. The raw response file hashes to `raw_response_hash`.
2. The think trace file + final output file are derivable
   from the raw response file by the documented parsing
   algorithm.
3. No content was substituted between capture and sealing.

#### 6.3 Future: Provider-Side Response Signing

When LLM providers begin offering response signatures
(D-1 §7, Future Work), the factory will:

1. Extract the provider signature from the API response
   headers.
2. Include the signature in the POST-SEAL metadata as a new
   field: `"provider_signature": str`.
3. The verification procedure (§9) will add a check:
   verify the provider signature against the provider's
   public key and the `raw_response_hash`.

This requires zero changes to the seal engine itself. The
provider signature is simply additional metadata in the
`SealPayload`. The seal engine hashes it along with
everything else.

### §7. The `seal_step` Helper Function

To avoid duplicating the seal sequence logic across every node,
all nodes call a single helper function.

```plaintext
pseudocode:

async def seal_step(
seal_engine: SealEngine,
state: FactoryState,
step_type: StepType,
content_file_paths: list[str],
metadata: dict,
chain_dir: str
) -> tuple[FactoryState, SealedStepReceipt]:
"""Execute a single seal step.
1. Increment state.step_index.
2. Compute content_hash over all content files

   (concatenated hashes if multiple files).

3. Construct SealPayload.
4. Write SealPayload to chain directory.

5. Call seal_engine.seal().
6. Update state.latest_hash and state.step_index.

7. Return updated state and receipt.

Args:
    seal_engine: The SealEngine instance.
    state: Current FactoryState (will be mutated).
    step_type: The StepType enum value for this step.
    content_file_paths: List of file paths whose
        content is being sealed. Their hashes are
        concatenated (sorted by path) to produce
        a single content_hash.
    metadata: Step-type-specific metadata dict.
    chain_dir: Path to the chain/ directory.

Returns:
    Tuple of (updated state, receipt).

Raises:
    SealError on any failure.
"""
```

```plaintext
**Content hash computation for multiple files:**

When a seal step covers multiple files (e.g., POST_GENERATOR
covers draft + think trace + inference record), the
content_hash is computed as:
content_hash = hash(
sorted_file_hash_1 + ":" +
sorted_file_hash_2 + ":" +
...
)
```
Where the file hashes are sorted by file path (lexicographic)
and concatenated with `:` separators. The concatenated string
is then hashed one final time to produce a single content_hash.

This ensures:
- The content_hash is deterministic regardless of the order
  files are listed.
- Adding or removing a file changes the content_hash.
- The individual file hashes are recorded in the metadata
  for granular verification.

#### 9.2 Verification Algorithm

The verification procedure is executed by `slop-cli verify`
(§2.1) and/or by a Python-side wrapper that invokes it and
parses the result. The algorithm:

```plaintext
pseudocode:

def verify_chain(chain_dir):
receipts = load_all_receipts_sorted_by_step_index(chain_dir)
  if len(receipts) == 0:
      return FAILURE("Empty chain")

  # Check genesis
  if receipts[0].step_type != "GENESIS":
      return FAILURE("First step is not GENESIS")
  if receipts[0].prev_hash != "":
      return FAILURE("Genesis prev_hash is not empty")

  # Verify each step
  for i, receipt in enumerate(receipts):
      # 1. Chain linkage
      if i > 0:
          if receipt.prev_hash != receipts[i-1].seal_hash:
              return FAILURE(f"Step {i}: prev_hash mismatch")

      # 2. Content integrity
      #    Re-compute the content hash from the referenced
      #    content files and verify it matches.
      #    (This requires the workspace to be intact.)
      expected_content_hash = recompute_content_hash(receipt)
      if expected_content_hash != receipt.content_hash:
          return FAILURE(f"Step {i}: content hash mismatch")

      # 3. Seal integrity
      #    Recompute the seal hash from the payload
      #    (prev_hash + serialized payload) and verify
      #    it matches receipt.seal_hash.
      expected_seal_hash = recompute_seal(receipt)
      if expected_seal_hash != receipt.seal_hash:
          return FAILURE(f"Step {i}: seal hash mismatch")

  # 4. Step index continuity
  for i in range(len(receipts) - 1):
      if receipts[i+1].step_index != receipts[i].step_index + 1:
          return FAILURE(f"Gap in step indices: "
                         f"{receipts[i].step_index} → "
                         f"{receipts[i+1].step_index}")

  # 5. Timestamp plausibility (D-1 §10)
  timestamp_report = check_timestamp_plausibility(receipts)

  # 6. If MANIFEST exists, verify output binding
  manifest_step = find_step_by_type(receipts, "MANIFEST")
  if manifest_step:
      verify_manifest_output_binding(manifest_step, chain_dir)

  return SUCCESS(timestamp_report)
  ```
#### 9.3 Timestamp Plausibility Checks

Per D-1 §10, the verification procedure must check that
inter-seal intervals are consistent with expected operations.

| Transition | Expected Minimum Interval | Suspicion Threshold |
|---|---|---|
| PRE_GENERATOR → POST_GENERATOR | 5 seconds (LLM inference) | < 1 second |
| PRE_VERIFIER → POST_VERIFIER | 3 seconds (LLM inference) | < 0.5 seconds |
| PRE_REVISER → POST_REVISER | 5 seconds (LLM inference) | < 1 second |
| TOOL_CALL → TOOL_CALL | 0.1 seconds (API roundtrip) | < 0.01 seconds |
| Any seal → next seal | 0 seconds | Negative (clock skew) |
| POST_VERIFIER → PRE_REVISER | 0 seconds (immediate routing) | > 3600 seconds (> 1 hour gap suggests manual intervention) |

**What plausibility checks DO NOT prove:** A fast seal interval
(e.g., 0.5 seconds between PRE_GENERATOR and POST_GENERATOR)
raises suspicion but does not constitute proof of forgery.
The operator might have a local model with extremely fast
inference. The check produces **warnings**, not failures.

**What plausibility checks DO prove:** A negative interval
(timestamp goes backward) proves either clock manipulation
or a reordering attack. This IS treated as a failure.

```plaintext
pseudocode:

def check_timestamp_plausibility(receipts):
suspicious = []
  for i in range(len(receipts) - 1):
      t_from = parse_iso(receipts[i].timestamp)
      t_to   = parse_iso(receipts[i+1].timestamp)
      delta  = (t_to - t_from).total_seconds()

      # Negative interval = FAILURE
      if delta < 0:
          return FAILURE(f"Timestamp regression at step "
                         f"{receipts[i+1].step_index}")

      # Check against expected minimums
      transition = (receipts[i].step_type,
                    receipts[i+1].step_type)
      expected_min = EXPECTED_MINIMUMS.get(transition, 0)

      if delta < expected_min:
          suspicious.append({
              "from_step": receipts[i].step_index,
              "to_step": receipts[i+1].step_index,
              "interval_seconds": delta,
              "expected_min_seconds": expected_min,
              "warning": f"Interval {delta:.3f}s is below "
                         f"expected minimum {expected_min}s "
                         f"for {transition[0]} → {transition[1]}"
          })

  return TimestampReport(
      all_monotonic = True,
      suspicious_intervals = suspicious
  )
```

### §10. Crash Recovery Protocol

Per D-1 §10 and D-1 Attack 4: "Crash recovery MUST re-verify
the full chain before resuming."

#### 10.1 Crash Detection

The factory detects a crash when:
1. An ARQ job is retried (ARQ sets a retry counter).
2. The `state.json` file exists but `state.status` is not
   a terminal state (`COMPLETED`, `FAILED`, `NO_OUTPUT`).
3. The chain directory contains seal receipts but the last
   receipt's step_index does not match `state.step_index`.

#### 10.2 Recovery Sequence
```plaintext

CRASH RECOVERY PROTOCOL


Load state.json from workspace.
If state.json does not exist:
→ This is not a recovery; it is a fresh start.
→ Abort recovery; re-run from genesis.


If state.status is terminal:
→ The run was already completed. No recovery needed.
→ Log and exit.


Verify the full chain:
report = await seal_engine.verify_chain(chain_dir)
If report.is_valid == False:
→ The chain is corrupted. DO NOT RESUME.
→ Set state.status = FAILED
→ Set state.last_error = f"Chain verification failed "
f"during crash recovery: {report.failure_reason}"
→ Write state.json
→ Raise CrashRecoveryError
→ The workspace is preserved for forensic analysis.


If the chain is valid:
→ Identify the last sealed step from the chain directory.
→ Compare last_sealed_step.step_index to state.step_index.
Case A: They match.
→ The crash occurred AFTER the last seal completed
but BEFORE the next phase began.
→ Resume at the beginning of the next node.
→ Log: "Resuming from step {N+1}"
Case B: last_sealed_step.step_index < state.step_index.
→ The crash occurred DURING a seal operation.
→ The state.json was updated but the seal was not
written (or was partially written).
→ ROLLBACK state.step_index to last_sealed_step.step_index.
→ ROLLBACK state.latest_hash to last_sealed_step.seal_hash.
→ Re-execute the interrupted node from its Phase 1
(PRE-SEAL).
→ Log: "Rolled back to step {last_sealed} and "
"re-executing interrupted node."
Case C: last_sealed_step.step_index > state.step_index.
→ This should be impossible (state is updated before
the seal). If it occurs, treat as corruption.
→ Set state.status = FAILED
→ Raise CrashRecoveryError


After recovery, continue the normal orchestration loop.
```

#### 10.3 Partial LLM Response Handling

If the crash occurred during Phase 2 (INFERENCE) of a node:
- The PRE-SEAL was completed (the intent is sealed).
- The LLM response may be partial, corrupt, or absent.
- The POST-SEAL was NOT completed.

Recovery action:
- Delete any partial response files in the drafts/ directory
  for the interrupted cycle.
- Re-execute Phase 2 (INFERENCE) from scratch.
- The re-executed inference will produce a DIFFERENT response
  (LLMs are stochastic). This is acceptable: the PRE-SEAL
  records the intent, and the POST-SEAL records whatever
  response was actually received.
- The chain honestly records: "We intended X (sealed), the
  first attempt was interrupted, and we received Y on the
  second attempt (sealed)."

**Cost of re-execution:** One additional LLM API call. This
is the price of crash safety. It is far cheaper than
restarting the entire run from genesis.

### §11. Provenance Disabled Mode

Per D-0 §8 and D-1 §10, provenance can be disabled for
development/testing by setting `config.enable_provenance = false`.

**Enforcement:** The environment variable
`ANTIPHORIA_I_UNDERSTAND_NO_PROVENANCE=true` must ALSO be set.
If `enable_provenance` is false but the environment variable
is not set, the factory refuses to start and prints:
ERROR: Provenance is disabled in configuration, but the
environment variable ANTIPHORIA_I_UNDERSTAND_NO_PROVENANCE
is not set to "true".
Disabling provenance removes all tamper-evidence and
auditability guarantees. The output will carry NO
cryptographic proof of how it was generated.
If you understand this and wish to proceed, set:
export ANTIPHORIA_I_UNDERSTAND_NO_PROVENANCE=true
For production use, set enable_provenance: true.
**When provenance is disabled:**
- The `seal_step` helper becomes a no-op that returns a
  dummy receipt with `seal_hash = "PROVENANCE_DISABLED"`.
- `state.latest_hash` is set to `"PROVENANCE_DISABLED"`.
- No chain/ directory is created.
- No manifest is compiled.
- The HAI Card includes a prominent warning:
  `provenance_status: "DISABLED — No cryptographic seals were
  generated for this run."`
- The output paper.md metadata header includes:
  `provenance_manifest: "NONE — Provenance was disabled."`

This mode exists solely for rapid development iteration.
It is never used for any output intended for publication
or external consumption.

### §12. Performance Budget

Seal operations must not dominate the runtime of the factory.
LLM inference calls typically take 10–300 seconds. Seal
operations must complete in under 1% of that time.

| Operation | Expected Latency | Acceptable Maximum |
|---|---|---|
| `slop-cli hash` (single file, < 1 MB) | < 5 ms | 50 ms |
| `slop-cli seal` (single step) | < 10 ms | 100 ms |
| `slop-cli verify` (30-step chain) | < 100 ms | 1 second |
| `slop-cli verify` (200-step chain) | < 500 ms | 5 seconds |

**Measured against a typical run:**

| Run Profile | Cycles | Seals | Est. Seal Time | Est. LLM Time | Overhead |
|---|---|---|---|---|---|
| Quick (1 cycle, 5 citations) | 1 | ~15 | ~150 ms | ~60 s | 0.25% |
| Normal (3 cycles, 10 citations) | 3 | ~55 | ~550 ms | ~300 s | 0.18% |
| Heavy (10 cycles, 15 citations) | 10 | ~170 | ~1.7 s | ~1500 s | 0.11% |

The provenance overhead is negligible. It never justifies
disabling provenance for performance reasons.

### §13. Error Scenarios and Handling

| Scenario | Detection | Response | Chain State |
|---|---|---|---|
| `slop-cli` binary not found | `FileNotFoundError` on subprocess | `FAILED` immediately. Cannot run without seal engine. | No chain created. |
| `slop-cli seal` returns exit code 1 (file not found) | `SealError(exit_code=1)` | `FAILED`. Log error. Preserve workspace. | Chain is valid up to the last successful seal. |
| `slop-cli seal` hangs (> 30s) | Subprocess timeout | `FAILED`. Kill subprocess. Log timeout. | Same as above. |
| Disk full during file write | `OSError` | `FAILED`. Log error. | Chain may be partially written. Recovery will detect corruption. |
| LLM API timeout during Phase 2 | LiteLLM retry exhaustion | Retry per LiteLLM config. If all retries fail: `FAILED`. PRE-SEAL is valid; no POST-SEAL. | Chain is valid up to PRE-SEAL. Recovery re-executes inference. |
| Instructor validation failure (Verifier) | `instructor.RetryError` | Write raw response. POST-SEAL with error metadata. Route to `human_rescue_queue`. | Chain is valid. The raw error response is sealed. |
| Chain verification failure during crash recovery | `verify_chain` returns `is_valid=False` | `FAILED`. Do NOT resume. Preserve workspace. | Chain is corrupt. Forensic inspection required. |
| State.json missing during crash recovery | `FileNotFoundError` | Cannot recover. Treat as a fresh run. Old workspace is orphaned. | New chain starts from genesis. |

**The golden rule:** If the seal engine reports any error,
the factory stops. It never attempts to "work around" a
seal failure by skipping the seal and continuing. A skipped
seal breaks the chain, and a broken chain is worse than
no chain (D-1 §6: it creates false confidence).

### §14. External Auditor Workflow

This section describes how a third party, with no access to
the factory's source code or runtime environment, verifies
a published artifact.

**What the auditor receives:**
1. The output paper (paper.md or paper.pdf).
2. The `chain/` directory (containing all seal receipts
   and the manifest.json).
3. The `drafts/` directory (containing all prompts,
   responses, and inference records).
4. The `citations/` directory (containing all tool results).
5. The `hai_card.md`.

**What the auditor does NOT receive:**
- The `slop-cli` binary (they install their own copy).
- The factory source code (they don't need it).
- API keys or model access.

**Verification steps:**
```plaintext
EXTERNAL AUDITOR WORKFLOW


Install slop-cli (or equivalent verification tool).


Run: slop-cli verify --chain-dir ./chain/
→ Confirms chain integrity: linkage, content hashes,
seal hashes, step continuity.
→ Confirms timestamp monotonicity.
→ Reports any suspicious timestamp intervals.


Inspect manifest.json:
→ Read configuration_overrides. Were any safety
settings weakened?
→ Read verdict_history. How many cycles? What were
the intermediate verdicts?
→ Read citation_summary. How many citations were
verified vs. not_found?
→ Read models_used. What models generated the content?
→ Read security_statement. Understand what the chain
does and does not prove.


Spot-check content integrity:
→ Pick a random POST_GENERATOR receipt.
→ Independently hash the corresponding draft file.
→ Confirm it matches the content_hash in the receipt.


Inspect the HAI Card:
→ Confirm human_review_status.
→ Confirm the number of cycles matches manifest.
→ Confirm no configuration_overrides are concealed.


Read the prompts:
→ Open drafts/cycle_01_generator_prompt.md.
→ Confirm the ResearchBrief is present and unmodified.
→ Open drafts/cycle_01_verifier_prompt.md.
→ Confirm citation check results are injected.


Assess the think traces (if captured):
→ Open drafts/cycle_01_generator_think.md.
→ Read the model's internal reasoning.
→ Evaluate whether the final output is consistent
with the reasoning trace.


Form an opinion:
→ The chain proves process provenance.
→ The auditor must independently assess scientific
correctness. The chain cannot do this for them.
```

**Tooling deliverable:** A simple HTML-based chain viewer
that automates steps 2–5 above. The auditor drags the
workspace directory into the viewer, and it renders a
visual timeline with expandable details. This is specified
in D-0 §9.4 and will be implemented in D-0 §13, Step 13.

### §15. Mapping to C2PA Content Credentials

The hash chain produced by the factory maps onto the
C2PA (Coalition for Content Provenance and Authenticity)
specification as follows:

| C2PA Concept | Factory Equivalent |
|---|---|
| **Asset** | The final `paper.md` file |
| **Claim** | The `ProvenanceManifest` |
| **Assertion: Action** | Each `SealedStepReceipt` (one action per seal step) |
| **Assertion: Creative Work** | The `ResearchBrief` (human input) |
| **Assertion: Data Hash** | The `content_hash` and `output_hash` fields |
| **Claim Signature** | The MANIFEST seal's `seal_hash` (and, in the future, an X.509 signature over the manifest) |
| **Ingredient** | Each external tool response (Crossref, Semantic Scholar, Tavily) sealed as TOOL_CALL |

**Current limitation:** The factory produces a C2PA-*compatible*
manifest structure but does NOT produce a cryptographically
signed C2PA manifest in the formal sense. Formal C2PA compliance
requires:
1. An X.509 certificate chain (the operator must obtain a
   code-signing certificate).
2. A COSE (CBOR Object Signing and Encryption) signature
   over the manifest.
3. Embedding the manifest as a JUMBF (JPEG Universal Metadata
   Box Format) box in the output file — which is defined for
   images and video, not Markdown or PDF.

**Phase 2 goal:** Implement formal C2PA signing using the
`c2pa-python` library. This requires the operator to provide
an X.509 certificate (self-signed is acceptable for open-source
use; CA-signed provides stronger trust). The manifest would be
embedded as a detached sidecar file (`.c2pa` extension) rather
than JUMBF-embedded, since our output format is Markdown/PDF.

**Aletheia comparison:** DeepMind's provenance approach for
Aletheia relied on institutional mechanisms rather than
cryptographic ones: "we e-mailed our solutions privately to the
FirstProof authors at 11:07pm PST on February 13" (Feng et al.,
2026a, §3). Our system replaces the trusted email with a
trustless hash chain. The email proves "DeepMind sent this at
time T." The hash chain proves "these specific operations
occurred in this specific order, and the output has not been
modified since." The hash chain is weaker on identity (who
ran it?) but stronger on process (what exactly happened?).
C2PA signing closes the identity gap.
