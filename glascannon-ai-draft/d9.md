### §1. Purpose and Scope

This is the final document in the specification suite. It
transforms the architecture defined in D-0 through D-8 into
a running system on real hardware.

Every preceding document assumed the system existed. This
document makes it exist.

**What this document decides:**
- The two deployment tiers: Local Development and Production.
- The exact directory structure of the repository.
- The Python environment and dependency management.
- The seal engine installation procedure.
- Environment variable contracts for API keys and configuration.
- The Docker Compose topology (Production tier).
- Persistent volume requirements.
- The CLI interface for running the factory.
- Operational procedures: starting, stopping, monitoring,
  upgrading, and backing up.
- Log management and observability.
- Cost estimation and budget management.

**What this document does NOT decide:**
- The internal implementation of nodes (D-0, D-3, D-4, D-5).
- The types and schemas (D-2).
- The prompts (D-3).
- The evaluation suite (D-8).
- The governance model (D-7).

**Key upstream constraints:**
- D-0 §10.1: "The system must be implementable on a single
  machine with Python 3.12+, an internet connection, and
  local disk storage. Phase 1 requires no Redis, no
  distributed queue, and no Kubernetes."
- D-0 §10.1: Phase 1 uses a hand-rolled async state machine.
  Phase 2 adds ARQ + Redis if justified by load.
- D-5 §2: The seal engine is an external binary (`slop-cli`).
- D-5 §11: Disabling provenance requires the environment
  variable `ANTIPHORIA_I_UNDERSTAND_NO_PROVENANCE=true`.
- D-8 §8: CI/CD pipeline requires E1 tests with zero network
  access and E2 tests with mock LLM calls.

### §2. Deployment Tiers

The factory supports two deployment tiers. They differ in
infrastructure complexity, not in functionality. The same
factory code runs in both tiers.

| Attribute | Tier 1: Local Development | Tier 2: Production |
|---|---|---|
| **Target** | A developer's laptop or a single VM | A persistent server or cloud instance |
| **Runtime** | Bare Python 3.12+ in a virtual environment | Docker Compose with 2–3 containers |
| **State persistence** | Local filesystem (`./workspaces/`) | Docker volume mounted to host or cloud storage |
| **Job queue** | None (single run, CLI invocation) | ARQ + Redis (Phase 2 only) |
| **Crash recovery** | Manual restart; `state.json` checkpoint | Automatic restart via Docker; `state.json` checkpoint |
| **Monitoring** | Terminal output + log files | Structured JSON logs + optional Streamlit dashboard |
| **Concurrent runs** | 1 (sequential) | 1 in Phase 1; 1-10 in Phase 2 (limited by API rate limits) |
| **Estimated setup time** | 15 minutes | 45 minutes |

**Tier 1 is the default.** Every developer and every CI
pipeline runs Tier 1. Tier 2 is deployed only when the
system must run unattended, handle multiple briefs in
sequence, or survive server restarts.

A practical middle ground also exists: Tier 1.5 is a
systemd-managed single-process deployment on a Linux server.
It uses the same bare-Python runtime as Tier 1 but runs under a
service manager for unattended execution. This is still Phase 1
architecture: no Redis, no queue, and only one active run at a
time.

- `src/antiphoria/types/human_rescue.py` contains `HumanRescueRequest` and `HumanRescueResponse`.
- `src/antiphoria/verification/tiers.py` holds tier ceilings and per-task method ceilings, while `verification/confidence.py` handles composition and floors.

- links or relative paths to the paper, HAI Card, manifest, and chain directory
- a summary of citation outcomes and human intervention events
- a distinct section for post-manifest chain extensions, if any

### §4. Python Environment

#### 4.1 Python Version

**Minimum: Python 3.12**

Python 3.12 is required for:
- `match/case` statements in the routing logic (available
  since 3.10, but 3.12 provides performance improvements).
- `type` statement for type aliases (3.12+).
- Improved `asyncio` performance.
- `tomllib` in the standard library (for reading
  `pyproject.toml` without external dependencies).

If community demand later justifies Python 3.11 support, the
only intentionally 3.12-specific language feature in v0.1 is
the `type` statement; it can be replaced with
`typing.TypeAlias` without architectural change.

#### 4.2 Dependency Management

Dependencies are managed via `pyproject.toml` using `pip`
and optionally `uv` (for faster resolution).

The repository also commits a lockfile (`requirements.lock`
or `uv.lock`) for reproducible installs. Local development may
use unconstrained resolution during experimentation, but Docker
images, CI, and published setup instructions MUST install from

#### 4.3 Why Each Dependency
| Dependency | Purpose | Why This One | Alternatives Rejected |
|---|---|---|---|
| litellm | Unified LLM API client | Standardizes OpenRouter, Gemini, Ollama to OpenAI format. Native async. Automatic retries. Fallbacks. (D-0 §2) | Raw httpx (too much boilerplate for each provider). openai SDK (single-provider). |
| instructor | Structured LLM output | Patches LiteLLM to guarantee Pydantic output. Automatic retries on validation failure. (D-0 §2, D-3 §4.5) | Manual JSON parsing + regex (brittle, per D-0 §2). |
| pydantic | Data validation | Runtime validation of LLM outputs and external inputs. JSON Schema generation for Instructor. (D-2) | dataclasses alone (no runtime validation). attrs (less ecosystem integration with Instructor). |
| httpx | HTTP client | Async-native. Used for Crossref, Semantic Scholar, Tavily API calls. | aiohttp (more complex API). requests (synchronous). |
| anyio | Async compatibility | Provides structured concurrency primitives. Works with both asyncio and trio. | Raw asyncio (less ergonomic for cancellation and timeouts). |
| click | CLI framework | Simple, well-documented, widely adopted. | argparse (more verbose). typer (adds a dependency on click anyway). |
| rich | Terminal output | Pretty-prints progress, tables, and errors. | Plain print (less readable for long runs). |
| arq | Async job queue (Phase 2) | Lightweight, pure-async Redis queue. (D-0 §2) | celery (too heavy, sync-first, per D-0 §2). |
Total dependency count: 7 core + 5 dev + 1 eval + 3 production = 16.
The project enforces a dependency budget: no new dependency may be added without TSC approval (D-7 §7.3). The approval requires justification of why an existing dependency or standard library module cannot serve the purpose.

Tavily calls are made directly with `httpx`; no Tavily-specific
SDK is required in v0.1.

The `.env` loader remains dependency-free in v0.1. Supported
syntax is intentionally narrow:
- `KEY=VALUE`
- `KEY="VALUE WITH SPACES"`
- blank lines and `# comment` lines

Unsupported syntax includes multiline values, interpolation,
and `export KEY=...`. System environment variables always win
over values from `.env`.

### §5. Seal Engine Installation
The seal engine (slop-cli or equivalent) is an external binary, not a Python package. It must be installed separately.
#### 5.1 Installation Script
The scripts/install_seal_engine.sh script handles installation for supported platforms.
```plaintext
SEAL ENGINE INSTALLATION

The script performs:

1. Detect platform (Linux x86_64, Linux arm64, macOS x86_64,

   macOS arm64).

2. Download the appropriate binary from the seal engine's

   release page.

3. Verify the binary's checksum (SHA-256).
4. If the project publishes detached signatures, verify the
   release signature against the seal engine maintainer key.
   Signature failures are warnings in v0.1; checksum failures
   are fatal.
5. Install to ~/.local/bin/ (or a user-specified path).

6. Verify: slop-cli --version
7. Add to PATH if not already present.

Supported platforms:

  - Linux x86_64
  - Linux arm64

  - macOS x86_64 (Intel)
  - macOS arm64 (Apple Silicon)

  - Windows x86_64 (via WSL2)

If the platform is not supported, the script prints
instructions for building from source.
```

Native Windows support is not planned for v0.1. Windows users
must run the factory through WSL2.

#### 5.2 Verifying the Seal Engine
After installation, the factory verifies the seal engine at startup:

```python
pseudocode:

  async def verify_seal_engine(cli_path: str) -> bool:
      """Called during 'antiphoria doctor' and at run start."""

      # 1. Binary exists and is executable
      if not shutil.which(cli_path):
          raise SealEngineNotFound(
              f"{cli_path} not found on PATH. "
              f"Run: bash scripts/install_seal_engine.sh"
          )

      # 2. Version check
      version = await run_subprocess(cli_path, "--version")
      log(f"Seal engine version: {version}")

      # 3. Functional check: hash a known string
      test_file = write_temp_file(b"antiphoria-test")
      hash_result = await run_subprocess(
          cli_path, "hash", "--file", test_file,
          "--algorithm", "sha256"
      )
      expected = (
          "f0fa8e0902e9785339d6abf0e78454c180df2904ff85b786"
          "55cd748ff1d19a13"
      )
      if hash_result.strip() != expected:
          raise SealEngineCorrupt(
              f"Hash mismatch. Expected {expected}, "
              f"got {hash_result.strip()}"
          )

      # 4. Functional check: seal and verify a 2-step chain
      ... (mini chain creation and verification)

      return True
```

`antiphoria doctor` also checks seal-engine version
compatibility against a factory-maintained tested range, e.g.
`>=0.3.0,<0.5.0`.

#### 5.3 Seal Engine Configuration
The seal engine path is configurable via environment variable:

| Variable | Default | Description |
|---|---|---|
| ANTIPHORIA_SEAL_CLI | slop-cli | Path to the seal engine binary. If just a name, resolved via $PATH. If a path, used directly. |
| ANTIPHORIA_SEAL_ALGORITHM | sha256 | Hash algorithm passed to the seal engine. Must be supported by the binary. |
| ANTIPHORIA_SEAL_TIMEOUT | 30 | Maximum seconds per seal operation before timeout (D-5 §3.3). |

#### 6.2 antiphoria run
The primary command. Takes a research brief and runs the full pipeline.

```plaintext
Usage: antiphoria run [OPTIONS] BRIEF_PATH

  Run the factory pipeline on a research brief.

  BRIEF_PATH: Path to a JSON file containing a ResearchBrief
  (D-2 §5). Use '-' to read from stdin.

Options:
  --workspace-dir PATH        Base directory for workspaces.
                              [default: ./workspaces]
  --config PATH               Path to a TOML config file overriding
                              FactoryConfig defaults.
  --generator-model TEXT      Override config.generator_model.
  --verifier-model TEXT       Override config.verifier_model.
  --max-cycles INT            Override config.max_total_cycles.
  --target-length INT         Override config.target_length_words.
  --confidence-threshold FLOAT
                              Override config.verifier_confidence_threshold.
  --no-provenance             Disable provenance sealing.
                              Requires ANTIPHORIA_I_UNDERSTAND_
                              NO_PROVENANCE=true env var.
  --no-citations              Disable citation checking.
  --no-think                  Disable <think> token capture.
  --skip-doctor               Skip pre-flight checks after an
                              explicit earlier doctor run.
  --verbose / --quiet         Log verbosity.
  --dry-run                   Validate the brief and config,
                              print the plan, but do not execute.
  -h, --help                  Show this message and exit.

Examples:
  ### Basic run with defaults
  antiphoria run brief.json

  ### Custom workspace and model
  antiphoria run brief.json \
    --workspace-dir /data/factory \
    --generator-model deepseek/deepseek-r1

  ### Dry run to validate configuration
  antiphoria run brief.json --dry-run

  ### Read brief from stdin (piped from another tool)
  cat brief.json | antiphoria run -
```

Run lifecycle:
1. Parse and validate BRIEF_PATH → ResearchBrief
2. Load and merge configuration (defaults → config file → CLI flags)

3. Generate run_id (UUID4)
4. Create workspace directory

5. Run 'antiphoria doctor' checks (seal engine, API keys)
6. If --dry-run: print plan and exit

7. Execute orchestrator (D-0 §4):

   Genesis → Generator → Verifier → [Reviser → Verifier]* → Finalize

8. Print summary to terminal
9. Exit with code 0 (success), 1 (FAILED), 2 (NO_OUTPUT),

   3 (AWAITING_HUMAN)

CLI-to-config mapping:
| CLI Flag | Config Field |
|---|---|
| `--generator-model` | `generator_model` |
| `--verifier-model` | `verifier_model` |
| `--max-cycles` | `max_total_cycles` |
| `--target-length` | `target_length_words` |
| `--confidence-threshold` | `verifier_confidence_threshold` |
| `--no-provenance` | `enable_provenance = false` |
| `--no-citations` | `enable_citation_checking = false` |
| `--no-think` | `capture_think_tokens = false` |

Exit codes:
| Code | Meaning | Terminal State |
|---|---|---|
| 0 | Run completed successfully | COMPLETED |
| 1 | Run failed (system error) | FAILED |
| 2 | Generator declared NO_OUTPUT | NO_OUTPUT |
| 3 | Run awaiting human rescue | AWAITING_HUMAN |
| 4 | Configuration error (invalid brief, missing API key) | — (never started) |
| 130 | User interrupt (`Ctrl+C`) | — (best-effort checkpoint write before exit) |
#### 6.3 antiphoria rescue

```plaintext
Usage: antiphoria rescue [OPTIONS] WORKSPACE_PATH

  Respond to a workspace in AWAITING_HUMAN status.

Options:
  --decision [approve_as_is|revise_manually|modify_config|abort]
                              Required rescue decision.
  --draft PATH                Required when decision=revise_manually.
  --reviewer-id TEXT          Required for approve_as_is and
                              revise_manually. Prefer ORCID.
  --comment TEXT              Optional explanation.
  --config PATH               Required when decision=modify_config.
                              Points to a TOML file whose diff is
                              recorded in the HUMAN_GATE receipt.
```

The command:
1. Validates that the workspace is in `AWAITING_HUMAN`.
2. Builds a `HumanRescueResponse`.
3. Seals it as a `HUMAN_GATE_FROM` step.
4. Resumes FINALIZE or the next automated step as appropriate.

| Disk space < 100 MB | Critically low disk | Fatal | Free disk space before running |

Implementation rule:
```python
for key, value in parse_env_file(".env"):
    if key not in os.environ:
        os.environ[key] = value
```

Factory behavior settings (models, cycle caps, thresholds) are
configured via the TOML file or CLI flags, not via environment
variables. This prevents accidental config leakage between runs
on the same machine.

Submitting a brief via the dashboard (Phase 2):
The Streamlit dashboard provides a web form where the user pastes a research brief. The dashboard serializes it as a ResearchBrief, validates it, and pushes a job to the Redis queue. The ARQ worker picks it up and runs the factory pipeline. The dashboard polls the workspace for state.json updates and displays progress in real time. If Redis is unavailable, the dashboard falls back to read-only workspace inspection mode.

Redis persistence in Phase 2 is restart-oriented, not archive-
oriented. The `redis_data` volume preserves in-flight queue
state across container restarts, but Redis contents are not part
of the long-term backup set because durable run state lives in
the workspaces themselves.
### §10. Persistent Volume Requirements
The workspace directory MUST survive container restarts. This is the most critical operational requirement.
| Deployment | Volume Strategy | Backup Strategy |
|---|---|---|
| Local dev | ./workspaces/ on local filesystem | Manual (developer's responsibility) |
| Docker (single server) | Bind mount to /srv/antiphoria/workspaces/ | Cron job: rsync to backup location daily |
| Docker (cloud VM) | EBS (AWS), Persistent Disk (GCP), or other block storage | Cloud provider snapshot schedule |
| Kubernetes (future only; unsupported in v0.1) | PersistentVolumeClaim with ReadWriteMany access | Volume snapshot CRD |
Minimum disk space per run:
| Run Profile | Workspace Size | Notes |
|---|---|---|
| Quick (1 cycle, 5 citations) | 500 KB – 2 MB | Mostly text files |
| Normal (3 cycles, 10 citations) | 2 MB – 5 MB | More drafts, more tool responses |
| Heavy (10 cycles, 15 citations) | 5 MB – 15 MB | Many revision cycles |
Workspace retention policy: Workspaces are never automatically deleted. The operator is responsible for archival and cleanup. A utility command for listing workspaces by age and status may be added in a future version:
antiphoria list-workspaces --older-than 30d --status COMPLETED

Prefer local SSDs or block storage for workspace volumes.
Network filesystems (NFS, EFS, CIFS) add per-file latency that
accumulates across seal-heavy runs.

#### 12.2 Budget Guardrails
The factory does not enforce hard spending limits (it cannot know exact pricing at runtime without querying provider APIs). However, it provides soft guardrails:

| Guardrail | Mechanism | Default |
|---|---|---|
| max_total_cycles | Caps the number of LLM call cycles | 10 |
| Token count logging | Every InferenceRecord includes exact token counts | Always on |
| HAI Card token summary | Models Used table shows total tokens per role | Always included |

#### 13.1 Starting a Run
### 1. Verify environment

### 2. Create a research brief

cat > brief.json << 'EOF'
{

  "domain": "extremal combinatorics",
  "constraints": "Target audience: advanced undergraduates.",

}
EOF

### 3. Run the factory

antiphoria run brief.json --verbose

### 4. Inspect the result

antiphoria inspect workspaces/{run_id}

### 5. Verify provenance

antiphoria verify workspaces/{run_id}
#### 13.2 Resuming a Crashed Run
### 1. Identify the crashed workspace

ls workspaces/

### → a1b2c3d4-e5f6-7890-abcd-ef1234567890/

### 2. Inspect the state

antiphoria inspect workspaces/a1b2c3d4-...

### → Status: GENERATING (interrupted)

### 3. Resume

antiphoria resume workspaces/a1b2c3d4-...

### The resume command will:

### - Verify the chain (D-5 §10.2)

### - Roll back to the last sealed step if necessary

### - Re-execute the interrupted node

### - Continue the pipeline normally
#### 13.3 Handling Human Rescue
When a run reaches AWAITING_HUMAN:
### 1. Inspect the run

antiphoria inspect workspaces/{run_id}

### → Status: AWAITING_HUMAN

### → Latest verdict: WRONG (3rd rejection)

### → Latest critique: "The proof relies on an invalid

### application of Cauchy-Schwarz..."

### 2. Read the latest draft

cat workspaces/{run_id}/drafts/cycle_03_reviser_output.md

### 3. Read the Verifier's critique

cat workspaces/{run_id}/drafts/cycle_03_verifier_output.json | python -m json.tool

### 4. Make a decision

### Option A: Approve as-is (you've reviewed and accept the draft)

antiphoria rescue workspaces/{run_id} \
  --decision approve_as_is \
  --reviewer-id "0000-0001-2345-6789" \
  --comment "Reviewed. The Cauchy-Schwarz application is
             actually valid in this context."

### Option B: Edit manually

cp workspaces/{run_id}/drafts/cycle_03_reviser_output.md \
   workspaces/{run_id}/drafts/human_edit.md

### ... edit human_edit.md ...

antiphoria rescue workspaces/{run_id} \
  --decision revise_manually \
  --draft workspaces/{run_id}/drafts/human_edit.md \
  --reviewer-id "0000-0001-2345-6789"

### Option C: Abort

antiphoria rescue workspaces/{run_id} \
  --decision abort \
  --comment "The brief is too ambitious for current models."
Each rescue decision is sealed as a HUMAN_GATE step (D-5 §5.5).

### (GitHub, Zenodo, institutional repository, etc.)

### 4. Update the HAI Card with the public URL

### (per D-6 §11, post-generation modification procedure)

### 1. Pull latest code

git pull origin main

### 2. Update dependencies

pip install -r requirements.lock
pip install -e ".[dev]"

### 3. Run tests

make test-unit      # E1

### Daily backup script (add to cron)

### !/bin/bash

The Redis data (restart-oriented queue state only; durable run
state already lives in the workspaces).
### §14. Security Hardening (Tier 2)
For production deployments, the following hardening measures are recommended:

| Measure | Description | Priority |
|---|---|---|
| Non-root container user | The Dockerfile creates and uses the antiphoria user. Never run the factory as root. | Critical |
| Read-only filesystem | Mount the application code as read-only. Only /data/workspaces/ and /tmp/ are writable. | High |
| Network policy | The factory container needs outbound HTTPS only (ports 443). No inbound ports. The dashboard container needs inbound 8501. | High |
| API key rotation | Rotate API keys quarterly. Use Docker secrets or a secrets manager (Vault, AWS Secrets Manager) in production. Do not use .env files in production. | High |
| Resource limits | Set memory: 2G and cpus: 2 limits on the worker container. LLM calls are I/O-bound, not CPU-bound; the factory should never need more than 2 cores. | Medium |
| Workspace permissions | chmod 750 on the workspace directory. Only the antiphoria user and the backup process should have access. | Medium |
| Log sanitization | Ensure API keys never appear in logs. The factory must redact any environment variable value that contains the substrings KEY, SECRET, or TOKEN before logging. | High |
| TLS for Redis | If Redis runs on a separate host, enable TLS. If Redis runs on the same host (Docker Compose network), TLS is optional. | Medium (Phase 2 only) |
| Dependency pinning | Build Docker images from `requirements.lock` / `uv.lock`, not floating dependency resolution. | High |
| Container image scanning | Scan the built image with Trivy, Docker Scout, or equivalent before deployment. | High |
| No new privileges | Add `no-new-privileges:true` to the production container security options. | Medium |
| AppArmor/Seccomp | Apply a restrictive profile in hardened deployments. | Medium |
### §15. Troubleshooting Guide
| Symptom | Likely Cause | Resolution |
|---|---|---|
| antiphoria doctor reports seal engine not found | slop-cli not installed or not on PATH | Run bash scripts/install_seal_engine.sh |
| SealError: exit code 1 during run | File not found by seal engine; usually a race condition or disk full | Check disk space. Check that the workspace directory exists and is writable. Inspect the SealError.command field for the exact CLI invocation. |
| SealError: exit code 99 | Internal seal engine error | Check seal engine version. Re-run antiphoria doctor. If persistent, file an issue against the seal engine. |
| instructor.RetryError on Verifier call | Gemini Flash returned unparseable JSON 3 times | Check the raw response in drafts/cycle_{N}_verifier_raw_error.md. The model may have changed behavior. Try updating the Verifier prompt (D-3 §4). The run is routed to human rescue. |
| Run stuck in GENERATING for > 10 minutes | LLM API timeout. LiteLLM is retrying. | Check network connectivity. Check OpenRouter status page. If persistent, the model may be down. The run will eventually fail with a LiteLLM timeout error. |
| CrashRecoveryError: Chain verification failed | The chain was corrupted between the crash and the recovery attempt. Possible causes: disk corruption, manual file editing, or D-1 Attack 4 (crash-and-resume manipulation). | Do NOT attempt to resume. Inspect the workspace forensically. The VerificationReport in the error message will indicate which step failed. Start a new run. |
| HAI Card missing from completed workspace | Bug in the finalization node | Check state.json for errors. The finalization node writes the HAI Card before the manifest seal (D-5 §8). If the manifest seal exists but the HAI Card does not, the finalization code has a bug. File an issue. |
| Token counts in HAI Card don't match expectations | LiteLLM token counting may differ from the model's actual tokenizer | This is a known limitation. LiteLLM estimates token counts; exact counts come from the API response. Check InferenceRecord.input_tokens and output_tokens against the API response metadata. |
| ANTIPHORIA_I_UNDERSTAND_NO_PROVENANCE not working | Env var typo or not exported | Ensure the variable is exported: export ANTIPHORIA_I_UNDERSTAND_NO_PROVENANCE=true. Ensure the spelling is exact (no trailing spaces). |
| Docker container OOM-killed | Run exceeded 2G memory limit | Increase the memory limit in docker-compose.yml. The most memory-intensive operation is the Verifier call with a very long draft (> 20K words). Consider increasing to 4G for heavy runs. |
| Crossref API returns 429 (rate limited) | Too many citation checks in rapid succession | The factory's httpx client includes retry logic with exponential backoff (D-4 §6.1). If persistent, add a User-Agent header with a mailto email to access the polite pool. |
### §16. The Makefile
Common commands are wrapped in a Makefile for convenience.

### Makefile

```makefile
.PHONY: help install install-dev test-unit test-integration
.PHONY: test-all lint typecheck doctor run benchmark
.PHONY: docker-build docker-run clean clean-all

PYTHON := python3.12
VENV := .venv
PIP := $(VENV)/bin/pip
PYTEST := $(VENV)/bin/pytest
MYPY := $(VENV)/bin/mypy
RUFF := $(VENV)/bin/ruff
CLI := $(VENV)/bin/antiphoria

help:  ## Show this help
	@grep -E '^[a-zA-Z_-]+:.*?## .*$$' $(MAKEFILE_LIST) | \
		awk 'BEGIN {FS = ":.*?## "}; {printf "  %-20s %s\n", $$1, $$2}'

install:  ## Install production dependencies
	@test -d $(VENV) || $(PYTHON) -m venv $(VENV)
	$(PIP) install -e .

install-dev:  ## Install all dependencies (dev + eval)
	@test -d $(VENV) || $(PYTHON) -m venv $(VENV)
	$(PIP) install -e ".[dev,eval]"
	bash scripts/install_seal_engine.sh

test-unit:  ## Run E1 unit tests
	$(PYTEST) tests/unit/ -x --timeout=60 -q

test-integration:  ## Run E2 integration tests (mock mode)
	$(PYTEST) tests/integration/ -x --timeout=900 -q

test-all:  ## Run E1 + E2
	$(PYTEST) tests/ -x --timeout=900

lint:  ## Run linter
	$(RUFF) check src/ tests/

typecheck:  ## Run type checker
	$(MYPY) src/antiphoria/ --strict

doctor:  ## Check system health
	$(CLI) doctor

run:  ## Run factory on a brief (usage: make run BRIEF=path/to/brief.json)
	$(CLI) run $(BRIEF) --verbose

benchmark:  ## Run E3 benchmark suites
	$(PYTHON) eval/run_suite_b.py
	$(PYTHON) eval/run_suite_v.py
	$(PYTHON) eval/run_suite_a.py
	$(PYTHON) eval/run_suite_t.py
	$(PYTHON) eval/generate_dashboard.py

docker-build:  ## Build Docker image
	docker compose -f docker/docker-compose.yml build

docker-run:  ## Run factory in Docker (usage: make docker-run BRIEF=path)
	docker compose -f docker/docker-compose.yml run factory run /briefs/$(notdir $(BRIEF))

clean:  ## Remove build artifacts and caches
	rm -rf __pycache__/ .mypy_cache/ .pytest_cache/
	find . -name "*.pyc" -delete

clean-all:  ## Remove caches and virtual environment
	$(MAKE) clean
	rm -rf .venv/
```
