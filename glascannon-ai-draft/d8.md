### §1. Purpose and Scope

This document defines how we **measure** whether the
slop-research-factory works. Not "works" in the sense of
"produces correct research" — we have been explicit throughout
D-0 to D-7 that it does not. "Works" in the sense of:

1. Does the provenance chain maintain integrity under all
   specified operating conditions?
2. Does the Verifier catch the classes of errors it claims
   to catch (D-4)?
3. Do the confidence ceilings hold empirically?
4. Does the crash-recovery protocol actually recover without
   chain corruption?
5. Do the prompts (D-3) produce structured, parseable output?
6. Does the HAI Card accurately reflect the run metadata?
7. Does the system degrade gracefully under adversarial
   conditions?

**What this document decides:**
- The three evaluation tiers: Unit Tests, Integration Tests,
  and System Benchmarks.
- Exact test specifications for every invariant checklist
  item defined in D-1 through D-7.
- The synthetic evaluation dataset: a corpus of research briefs,
  known-good drafts, known-bad drafts, and adversarial inputs
  used for regression testing.
- The Verifier calibration protocol: how to measure whether
  the Verifier's confidence scores correspond to actual
  accuracy.
- The provenance stress test suite.
- The human evaluation protocol for periodic quality audits.
- CI/CD integration requirements.

**What this document does NOT decide:**
- Prompt text (D-3).
- Infrastructure deployment (D-9).
- Pass/fail thresholds for publication readiness (that is a
  human judgment, per D-7 §4.2).

**Key upstream constraints:**
- D-0 §11: Defines **7** success criteria (SC-1–SC-7). Each must
  have a corresponding test in this document (see traceability
  table below). If D-0 §11 or this matrix changes, update both
  in lockstep.
- D-3 §8.1: Defines 6 prompt evaluation tests (T1–T6).
  Each must be operationalized here.
- D-4 §3: Defines confidence ceilings per tier. The
  calibration protocol must verify these hold empirically.
- D-5 §16: Defines **18** provenance invariants as of D-5
  v0.1. Each must have an automated test. If D-5 is amended
  with additional invariants, this count and the test matrix
  must be updated in lockstep.
- D-6 §12: Defines **16** HAI Card invariants (including
  dimensional confidence breakdown and post-manifest
  verification expectations). Each must have an automated test.
- D-7 §16: Defines **15** governance invariants. Each must have
  an automated check.

**D-0 §11 success criteria → tests (coverage obligation):**

| D-0 criterion (paraphrased) | Covering tests / artifacts |
|---|---|
| SC-1: Complete provenance chain for every terminal run | E2-P01–P14, E2-MF01–MF08, E2-WS10 |

### §2. The Three Evaluation Tiers

The factory's evaluation infrastructure is organized into
three tiers, each with different execution frequency,
resource requirements, and failure consequences.

| Tier | Name | Frequency | LLM Calls Required | Duration | Failure Consequence |
|---|---|---|---|---|---|
| **E1** | Unit Tests | Every commit (CI) | Zero | < 60 seconds | PR blocked |
| **E2** | Integration Tests | Every PR to main (CI) | 2–5 (mocked or real) | < 15 minutes | PR blocked |
| **E3** | System Benchmarks | Weekly (scheduled CI) or manual | 50–200 (real API calls) | 2–8 hours | Quality report published; no block |

**Optional sub-tier E1.5 (local integration, no mocks):** Some
checks need `slop-cli` and a real filesystem but no LLM and no
network (e.g. chain verification helpers). Teams MAY run these
on every commit when the binary is present; CI SHOULD run them
whenever the seal engine install step succeeds. They are not a
substitute for full E2 mock integration.

| Sub-tier | Name | Requirements | Typical trigger |
|---|---|---|---|
| E1.5 | Seal-chain smoke | `slop-cli`, temp dirs | Every commit (if binary present) |

**Why E3 does not block PRs:** System benchmarks involve real
LLM API calls, which are stochastic, slow, and expensive. A
benchmark failure may indicate a model degradation (OpenRouter
changed the DeepSeek-R1 version), not a code regression. E3
failures trigger investigation, not automatic rejection.

**E3 duration (order-of-magnitude arithmetic):** Suite B is
dominant. Example: 12 briefs × ~3 cycles × (~1 Generator or
Reviser + 1 Verifier + 1 Citation Extractor) ≈ 12 × 3 × 3 ≈ 108
LLM calls, plus Suite V (~7 Verifier-only) and small Suite A/T
overhead → **roughly 120–170 LLM calls** per full E3 week. At
30–120 s per call (think-heavy Generator/Reviser), that is about
**1.5–6 hours** of LLM wall time before tool I/O; **2–8 hours**
total remains a reasonable bracket.

**E3 cost (order-of-magnitude):** Using D-9-style token
estimates (~$0.04–$0.05 per Generator/Reviser-equivalent cycle
and ~$0.004 per Verifier+citation-extractor cycle), one full E3
run is on the order of **$50–$150** depending on think-token
length and brief difficulty. At weekly cadence this is **~$2.5k–
$8k/year**; the schedule MAY be reduced to biweekly/monthly if
funding requires it.

### §4. Tier E2: Integration Tests

Integration tests verify interactions between components.
They require the filesystem (temporary directories), and
some require LLM API calls (real or mocked).

#### 4.1 Mock vs. Real LLM Calls

E2 tests default to **mock LLM responses**. A set of
pre-recorded response fixtures is stored in the repository.
These fixtures were captured from real API calls and
saved as JSON files.

Fixture structure:
tests/ fixtures/ llm_responses/ generator/ brief_topology_response.json # Full response including think tokens brief_numbertheory_response.json brief_impossible_no_output.json verifier/ draft_good_correct.json # VerifierOutput with CORRECT verdict draft_flawed_fixable.json # VerifierOutput with FIXABLE verdict draft_terrible_wrong.json # VerifierOutput with WRONG verdict draft_malformed_retry.json # Unparseable response (triggers retry) reviser/ revision_after_fixable.json revision_after_wrong.json citation_extractor/ extraction_10_citations.json extraction_0_citations.json tool_responses/ crossref/ doi_valid.json doi_not_found.json doi_metadata_mismatch.json semantic_scholar/ paper_found_with_abstract.json paper_not_found.json tavily/ search_results_normal.json search_results_empty.json

**To run E2 with real API calls** (for fixture refresh or
pre-release validation):
ANTIPHORIA_E2_REAL_API=true pytest tests/integration/

This requires valid API keys in environment variables.
Real-API E2 runs are NOT part of standard CI; they are
run manually before releases.

**Fixture metadata:** Each fixture SHOULD include `_fixture_version`,
`_fixture_captured_date`, and `_fixture_model` (or equivalent) so
schema drift is detectable. A validation test MUST reject fixtures
whose version is older than the supported schema.

**Raw API payloads:** For pipeline tests (D-5 §6), store companion
`raw_response_*.bin` or `.json` fixtures for at least one happy-path
and one error path per node type.

#### 5.3 Benchmark Suite V: Verifier Calibration

The Verifier calibration benchmark measures whether the Verifier's self-reported confidence scores correspond to actual accuracy.

**Small-N protocol note:** Quintile-binned calibration error is **not** meaningful at N=7. For the current dataset size, report **Spearman ρ** (Verifier confidence vs. human quality rank or binary correctness) alongside legacy metrics; treat numeric targets as **version-to-version trends**, not rigorous hypothesis tests, until N≥30.

Method:
Take the known-good drafts (D01, D02) and the known-bad drafts (D03–D07).
Run each through the Verifier node ONLY (bypass Generator).
Record the Verifier's verdict and all confidence scores.
Compare to ground truth:

| Draft | Ground Truth | Expected Verdict | Expected Confidence |
|---|---|---|---|
| D01 (human-written, good) | CORRECT | CORRECT | HIGH |
| D02 (verified correct) | CORRECT | CORRECT | HIGH |
| D03 (hallucinated citations) | WRONG or FIXABLE | FIXABLE minimum | citation_accuracy LOW |
| D04 (logical fallacy) | WRONG | WRONG or FIXABLE | logical_soundness LOW |
| D05 (scope violation) | FIXABLE | FIXABLE minimum | scope_compliance LOW |
| D06 (trivial restatement) | Correct but unoriginal | CORRECT or FIXABLE (novelty may be low; not a correctness failure by itself) | novelty typically LOW |
| D07 (all unverified tags) | FIXABLE | FIXABLE minimum | citation_accuracy LOW |
| D08 (plausible wrong, when added) | WRONG | Should not be CORRECT with high confidence | Any (flags “confident wrong”) |

Compute calibration metrics:

| Metric | Definition | Target |
|---|---|---|
| Verdict accuracy | % of drafts where the Verifier's verdict matches ground truth (CORRECT for good, WRONG/FIXABLE for bad) | > 70% |
| Confidence calibration error | (Legacy) MAE binned into quintiles — use only when N is large enough; else omit or mark “not computed” | < 0.2 |
| Spearman ρ (confidence vs. quality) | Rank correlation vs. human score or binary correctness | Track trend |
| False approval rate | % of known-bad drafts that receive CORRECT verdict | < 20% |
| False rejection rate | % of known-good drafts that receive WRONG verdict | < 20% |
| Citation detection rate | % of hallucinated citations in D03 correctly flagged as NOT_FOUND by tool checks | > 80% |

The 70% verdict accuracy target is deliberately low. We do not expect the Verifier to be highly accurate. We expect it to be better than random. Per D-4 §2, Principle 1: "The Verifier catches some errors, not all errors." The benchmark exists to measure HOW MANY errors it catches, not to assert that it catches all of them.

**Calibration failure response:** If the false approval rate exceeds 30% or the citation detection rate drops below 60%:

- File an issue tagged `verifier-degradation`.
- Review D-3 Verifier prompt for potential improvements.
- Check whether the Verifier model version has changed (OpenRouter model updates).
- Update D-4 confidence ceilings if empirical evidence contradicts the current ceilings.

### §6. The Human Evaluation Protocol

Automated benchmarks measure system behavior. They cannot measure output quality in any meaningful sense. Per D-7 §4.2: "the Level of Mathematical Significance can only be evaluated by mathematicians."

The project requires periodic human evaluation to:

Ground-truth the Verifier's assessments.
Identify blind spots not covered by known-bad drafts.
Track quality trends over time.
Provide calibration data for confidence score adjustment.

#### 6.1 Evaluation Frequency

Human evaluation is performed:

At every minor release (v0.x → v0.y): 3 briefs, evaluated by 1 reviewer.
At every major release (v0.x → v1.0, etc.): All 12 briefs in the dataset, evaluated by 2–3 independent reviewers.
On demand when a community member identifies a potential quality regression.

**On-demand trigger:** File a GitHub issue with label `quality-concern`, including `run_id` and a short description. A TSC member SHOULD schedule human evaluation within **14 days**.

#### 6.2 The Human Evaluation Rubric

Each factory output is evaluated on **six** dimensions using a 4-point scale. Five mirror the Verifier's confidence dimensions (D-2 §8.4); **Scientific Value** has no Verifier twin but informs AMRL significance (D-6 §8).

| Dimension | 1 (Poor) | 2 (Below Average) | 3 (Adequate) | 4 (Good) |
|---|---|---|---|---|
| Logical Coherence | Argument is incoherent or self-contradictory. | Argument has multiple logical gaps or unstated assumptions. | Argument is mostly coherent with 1–2 minor gaps. | Argument is clear, well-structured, and logically sound. |
| Citation Quality | Most citations are hallucinated or irrelevant. | Several citations are hallucinated; others are real but poorly attributed. | Most citations are real and relevant; 1–2 are questionable. | All citations are real, relevant, and properly attributed. |
| Mathematical/Technical Rigor | Major errors in reasoning or computation. | Several minor errors or unjustified claims. | Mostly rigorous with 1–2 informal steps. | Fully rigorous within the scope of the brief. |
| Brief Compliance | Draft ignores the thesis or violates multiple constraints. | Draft partially addresses the thesis or violates 1–2 constraints. | Draft addresses the thesis and respects most constraints. | Draft directly addresses the thesis and respects all constraints. |
| Readability | Poorly formatted, confusing structure, broken LaTeX. | Readable but with formatting issues or unclear passages. | Well-structured and readable with minor formatting issues. | Publication-quality formatting and clear prose. |
| Scientific Value | No scientific content; trivial or fabricated. | Some valid content but mostly known results without insight. | Useful synthesis or minor observations. | Insight or connection a researcher might find valuable. |

**Readability vs. automation:** FP-1–FP-5 counts (TODO markers, LaTeX errors, `[UNVERIFIED]` tags) SHOULD be recorded next to human Readability for correlation analysis.

Scoring: Each dimension receives a score of 1–4. The overall human score is the unweighted mean (range 1.0–4.0).

**Inter-rater protocol:** Reviewers evaluate **independently** (no visibility into others' scores). Report Cohen's κ on categorical verdict agreement and Pearson's r on dimensional scores. If κ < 0.4, refine rubric wording. Final published scores are the **mean** across reviewers.

Comparison to Verifier assessment: For each dimension, the human score is compared to the Verifier's corresponding confidence score (mapped to the 1–4 scale):
Verifier confidence → Human scale mapping:

```plaintext
  0.0–0.25 → 1
  0.25–0.5 → 2
  0.5–0.75 → 3
  0.75–1.0 → 4
```

The difference (human_score - mapped_verifier_score) is the Verifier bias for that dimension:
Positive bias = Verifier was too pessimistic (output is better than the Verifier thought).
Negative bias = Verifier was too optimistic (output is worse than the Verifier thought).
Zero = Verifier is calibrated.

All prompts and responses (to understand what the LLM was asked and what it produced).
The Verifier's VerifierOutput (to compare against their own assessment).
The citation check results (to verify whether tool-grounded checks caught the issues they identified).
The <think> traces (to understand the model's reasoning process).
The provenance chain guarantees that the workspace the evaluator reviews is the same workspace the factory produced. This is the chain's primary value in the evaluation context: it prevents the operator from cherry-picking or editing the artifacts before evaluation.

If the evaluator finds workspace files with **no matching seal** (or seals with no matching file), report it as a **provenance integrity** issue, not only a quality concern.

| Suite B: mean total_cycles | … | … | → |
| Suite B: NO_OUTPUT false negative count | … | … | ↓ |

Trend symbols: ↑ = improved, → = unchanged, ↓ = degraded.

#### 7.2 Regression Detection

A regression is detected when any of the following occur between consecutive versions:

| Condition | Severity | Response |
|---|---|---|
| Suite B: chain valid drops below 100% | Critical | Block release. The provenance system has a bug. |
| Suite V: false approval rate increases by > 10 percentage points | High | File verifier-degradation issue. Investigate prompt and model changes. |
| Suite V: citation detection rate drops below 60% | High | Investigate Crossref/Semantic Scholar API changes. |
| Suite V: mean Verifier confidence rises > 0.1 while human mean score is flat or down | High | Overconfidence drift — investigate calibration and ceilings. |
| Suite A: any adversarial test causes a crash | High | Fix before release. |
| Suite T: seal overhead exceeds 1% | Medium | Investigate seal engine performance. |
| Suite B: mean total_tokens up > 50% version-to-version | Medium | Likely model/provider change; update cost assumptions. |
| Human eval: mean overall score drops by > 0.5 | Medium | Investigate model version changes and prompt drift. |
| Human eval: mean Verifier bias becomes more negative by > 0.5 | Medium | The Verifier is becoming more over-optimistic. Tighten calibration. |

**Optional release gate:** E3 remains non-blocking for most metrics, but **chain_valid < 100%** MAY be treated as a **hard gate** (same severity as §7.2 Critical) so provenance integrity cannot ship regressed.

### §8. CI/CD Integration

#### 8.1 Pipeline Structure
GitHub Actions (or equivalent CI):

```plaintext
  on: push to any branch
    → E1 (Unit Tests)
       pytest tests/unit/ -x --timeout=60
       MUST PASS for green build.

  on: pull_request to main
    → E1 (Unit Tests)
    → E2 (Integration Tests, mock mode)
       pytest tests/integration/ -x --timeout=900
       MUST PASS for merge.

  on: schedule (weekly, Sunday 02:00 UTC)
    → E3 Suite B (12 briefs, real API)
    → E3 Suite V (7 drafts, real API)
    → E3 Suite A (5 adversarial, real API)
    → E3 Suite T (timing measurements)
    → Generate dashboard.md
    → Commit results to eval/results/{version}/
       DOES NOT BLOCK. Publishes results only.
    → If this job fails 2 weeks in a row, open issue `e3-failure` and notify TSC.

  on: schedule (daily, 03:00 UTC)  [optional]
    → E2 integration with ANTIPHORIA_E2_REAL_API=true
    → On failure: issue tagged api-regression

  on: release tag (v*.*.*)
    → E1 + E2 (MUST PASS)
    → E3 all suites (results published, DOES NOT BLOCK)
    → Human evaluation triggered (manual, 14-day window)
```

#### 8.2 Secret Management
E3 benchmarks require API keys for:

OpenRouter (Generator, Reviser: DeepSeek-R1)
Google AI (Verifier: Gemini Flash)
Semantic Scholar API
Crossref (polite pool, no key required but mailto header)
Tavily Search API
These are stored as CI secrets and injected as environment variables. They are NEVER committed to the repository.

E1 and E2 tests require NO API keys (all LLM calls are mocked in E2 default mode).

#### 8.3 Test Isolation

E1 tests use no filesystem (or temporary directories only).
E2 tests create temporary workspaces in /tmp/antiphoria_test_* and clean up after themselves.
E3 tests create persistent workspaces in eval/results/{version}/workspaces/ for post-mortem analysis.
E2 and E3 tests that invoke slop-cli require the binary to be available on the CI runner. The CI pipeline installs it in a setup step.

**CI policy:** On **merge-blocking** jobs, if `slop-cli` is missing, the job MUST **fail** (or use an explicit `ALLOW_SKIP_PROVENANCE=true` escape hatch for fork PRs only). Silent skip with only a log line is **not** allowed for main-branch protection — provenance tests are too central to mask.

### §9. The Evaluation Feedback Loop
The evaluation infrastructure feeds back into the system design through a defined process:
```plaintext
┌─────────────┐
│ E3 Benchmarks│──── results ────▶ ┌────────────────┐
│ (weekly)     │                    │ Quality        │
└──────────────┘                    │ Dashboard      │
                                    │ (dashboard.md) │
┌──────────────┐                    └───────┬────────┘
│ Human Eval   │──── scores ─────▶          │
│ (per release)│                            │
└──────────────┘                            ▼
                                    ┌────────────────┐
                                    │ Regression     │
                                    │ Detection      │
                                    └───────┬────────┘
                                            │
                         ┌──────────────────┼──────────────────┐
                         ▼                  ▼                  ▼
                 ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
                 │ D-3 Prompt    │ │ D-4 Confidence│ │ D-5 Seal      │
                 │ Revision      │ │ Ceiling Update│ │ Engine Fix    │
                 └───────┬───────┘ └───────┬───────┘ └───────┬───────┘
                         │                 │                  │
                         ▼                 ▼                  ▼
                 ┌─────────────────────────────────────────────────┐
                 │ New version with updated prompts / ceilings     │
                 │ → Re-run E3 → Update dashboard → Compare        │
                 └─────────────────────────────────────────────────┘
```

The key discipline: Prompt changes (D-3), ceiling changes (D-4), and architecture changes (D-0/D-5) are never made without re-running the E3 benchmarks. The dashboard provides the empirical basis for evaluating whether a change was an improvement or a regression.

**Defining “improvement” at small N:** A change is an improvement only if (a) no E1/E2 regressions, (b) Suite V verdict accuracy does not decrease, and (c) at least one headline E3 metric moves by at least **one countable unit** on the benchmark size (e.g. one additional correct verdict out of 7 ≈ 14 pp — coarse but honest at N=7).

**Closing the loop:** Investigations that find a new failure class MUST add a brief, draft, or automated test to `eval/` / E1–E2 before the issue is closed.

The anti-pattern to avoid: Changing the prompts based on a single anecdotal observation ("it produced a bad draft on topic X, so I tweaked the prompt"). All prompt changes must be justified by benchmark data showing improvement on the full suite, not just the motivating case. Per D-3 §8.2: "Prompt engineering is empirical; the rationale for changes must be preserved."

### §10. Relationship to Aletheia's Evaluation Methodology
The Aletheia papers describe evaluation methodologies that we cannot replicate but can learn from.

| Aletheia Evaluation | Our Equivalent | Gap |
|---|---|---|
| Expert panel evaluation (3–7 named mathematicians per problem) | Human Evaluation Protocol (§6) with 1–3 reviewers | Our reviewers may not be domain experts at the level of Aletheia's panelists. The rubric (§6.2) is coarser than expert mathematical judgment. |
| FutureMath Basic benchmark (PhD-level exercises with known solutions) | Suite V: Verifier Calibration (§5.3) with known-good and known-bad drafts | FutureMath has 882 carefully curated problems. Our dataset has 12 briefs and 7 calibration drafts. Scale is incomparable. |
| IMO-ProofBench (Olympiad problems with formal solutions) | No equivalent. Would require Lean 4 integration. | This is the single largest evaluation gap. Formal verification would upgrade all LM-* tasks from T3 to T1. |
| "Best of 2" selection with expert preference judgment | Not applicable (we do not select between multiple runs) | D-1 §6, Non-Defense 3 (selective run disclosure) addresses this differently — by requiring disclosure of all runs, not selection of the best. |
| Conditional accuracy at different computational budgets (Feng et al., 2026b, Figure 3) | Suite T plus optional Suite B add-on (§5.2) | Vary `max_total_cycles` on 1–2 briefs in E3 to estimate a crude scaling curve; cheap relative to full cross-compute sweeps. |
| Self-filtering accuracy ("No solution found" rate vs. actual difficulty) | Suite B tracks no_output_triggered per brief | Dataset should add **multiple** expected-NO_OUTPUT briefs (§5.1) so this proportion becomes measurable. |

### §11. Community Contribution to Evaluation
The evaluation dataset and human evaluation pool are the components most amenable to community contribution.

#### 11.1 Contributing Briefs
Any community member may propose new evaluation briefs by filing a PR that adds a B*.json file to eval/briefs/.
Acceptance criteria for new briefs:
The brief must be a valid ResearchBrief (D-2 §5).
The PR must include _eval_* metadata fields specifying expected behavior.
The brief must cover a domain or scenario not already represented in the dataset.
The brief should not require access to proprietary data or tools.
The brief should be reproducible: two runs on the same brief should produce structurally similar (not identical) output.
#### 11.2 Contributing Known-Good/Bad Drafts
Known-good drafts (for Verifier calibration) must be:

Written or verified by a human with domain expertise.
Accompanied by a brief explanation of why the draft is correct.
Reviewed by 1 TSC member.
Known-bad drafts must be:

Constructed with specific, documented flaws.
Accompanied by a list of the injected flaws and their locations.
Classified by which Verifier tasks should catch them:
D03_hallucinated_citations.md → should be caught by CV-1, CV-3 (T1 deterministic tasks).
D04_logical_fallacy.md → should be caught by LM-3 (T3 LLM task). May not be caught.
D05_scope_violation.md → should be caught by SC-2 (T3 LLM task). May not be caught.
#### 11.3 Contributing Human Evaluations
Any community member may contribute human evaluations for published benchmark runs. The evaluation must:

Follow the rubric (§6.2).
Reference a specific run_id and factory version.
Be filed as a PR adding a JSON file to eval/human_evaluations/{version}/.
Include the evaluator's identifier (name, ORCID, or pseudonym).
The project specifically invites evaluations from:

Domain experts (mathematicians, scientists) who can assess technical rigor.
Non-experts (students, journalists) who can assess readability and whether the output "feels like research" to a non-specialist. This is valuable data: if non-experts rate factory output highly on the rubric, it confirms the "slop" hypothesis — the output resembles research without being research.

**Adversarial inputs:** Community members MAY propose `A*.md` or config fixtures via PR with (a) attack intent, (b) expected non-crash and **chain validity**, (c) mapping to a D-1 attack id.

**Credit:** Evaluators contributing ≥3 recorded evaluations SHOULD be listed in `CONTRIBUTORS.md` (name / ORCID optional).

### §12. Long-Term Evaluation Goals

These are evaluation capabilities the project aspires to but cannot implement in v0.1.

| Goal | Dependency | Impact |
|---|---|---|
| Formal verification of mathematical claims via Lean 4 | Lean 4 integration (D-4 §15) | Would provide T1-level ground truth for mathematical rigor. Would make Suite V calibration definitive rather than approximate. |
| A/B testing of prompts on the full benchmark suite | Automated A/B framework with statistical significance testing | Would replace anecdotal prompt tuning with rigorous comparison. |
| Cross-model benchmarking | LiteLLM model swap (env e.g. `ANTIPHORIA_E3_GENERATOR_MODEL`) | **Phase 1 optional** — no code change required; cost is API spend only. |
| Scaling-law experiments | Vary `max_total_cycles` on the same brief (see §5.2) | **Phase 1 optional** — small add-on to Suite B; full 1–50 sweep remains a stretch goal. |
| Blind evaluation | Evaluators assess factory output alongside human-written drafts without knowing which is which | Would measure whether the "slop" assessment is empirically justified. If evaluators cannot distinguish factory output from human output, the project has either succeeded beyond expectations or the evaluation rubric is too coarse. |
| Longitudinal quality tracking | Run the same 12 briefs monthly with the same configuration, tracking quality over time as models update | Would detect both improvements (model upgrades) and regressions (model version changes, API behavior shifts). |
| Multi-language evaluation | Non-English briefs if/when supported end-to-end | Track rubric scores and tool-check behavior across languages. |

### §13. Test Implementation Priority
For a coding assistant implementing the test suite, the following priority order ensures that the most critical tests are written first.

| Priority | Test Group | Count | Why First |
|---|---|---|---|
| P0 | E1-S01–S23 (State schema) | 23 | Every other component depends on correct state types. |
| P1 | E1-R01–R10 (Routing logic) | 11 | Incorrect routing is the highest-impact logic bug. A draft routed to CORRECT when it should be FIXABLE is a false approval. |

| P4 | E1-V01–V20 (Deterministic verification) | 20 | T1 tasks are the most reliable checks. They must work perfectly. |

| P6 | E2-CR01–CR07 (Crash recovery) | 7 | Crash recovery with chain corruption is a critical failure mode (D-1 Attack 4). |
