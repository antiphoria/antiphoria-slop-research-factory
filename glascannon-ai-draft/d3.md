### §1. Purpose and Scope

This document specifies the exact system prompts, user message
templates, and structured output instructions for all three
cognitive nodes in the factory: the Generator, the Verifier, and
the Reviser.

**What this document decides:**

- The full text of every system prompt.
- The user message template for every node, including variable
  interpolation points.
- The Instructor/Pydantic coercion strategy for the Verifier's
  structured output.
- The exact mapping between Aletheia's published Verification-
  and-Extraction prompt (Feng et al., 2026a, Appendix A) and
  our adapted implementation.
- How `<think>` reasoning tokens are handled in prompt context.

**What this document does NOT decide:**

- Which external tools the Verifier calls or how they are
  implemented (D-4). D-3 specifies only how tool results and
  extraction outputs flow into prompts.
- How sealing integrates with node execution (D-5).
- Model selection or routing infrastructure (D-0, D-9).

**Key constraints from upstream documents:**

- D-1 §10: "The Verifier system prompt must instruct the model
  to output graduated confidence, not binary verdicts. The
  prompt must explicitly state: 'You are not the final arbiter
  of scientific truth. You are a pre-screening filter.'"
- D-2 §8.4: The Verifier must produce a `VerifierOutput` Pydantic
  object with the exact schema defined there.
- D-0 §4: The Reviser receives the same model as the Generator
  but with a dynamically injected critique.

### §2. Design Principles for All Prompts

#### 2.1 Direct Lineage from Aletheia

The prompt architecture is derived directly from two sources in
the Aletheia papers:

| Source | What We Extract | Where It Goes |
|---|---|---|
| Feng et al. 2026a, Appendix A (the Verification-and-Extraction prompt) | The three-phase structure: (1) Independent Verification → (2) Critique → (3) Verdict → Resolution | The Verifier system prompt, adapted for general science |
| Feng et al. 2026b, §2.2 (on separating generation from verification) | The principle that "decoupling a reasoning model's final output from its intermediate thinking tokens, and adding well-chosen prompt scaffolding, enables the model to recognize flaws it initially overlooked during generation" | The architectural separation of Generator and Verifier; the Reviser receiving only the Verifier's structured critique, never its own prior think trace |
| Feng et al. 2026b, §2.3 (on tool use and hallucination) | "Simply granting internet access is insufficient; however, when our model was extensively trained for tool use, the integration of Google Search and web browsing led to a substantial reduction in obvious citation hallucinations" | The Verifier's tool-grounding preamble and the Generator's explicit instruction to flag uncertain citations |
| Feng et al. 2026a, §2 (on autonomy and correctness interpretation) | "We interpreted 'Correct' as meaning 'publishable after minor revisions, within the established range of the peer review process'" | The Verifier's CORRECT verdict calibration |

#### 2.2 Structural Rules for All Prompts

1. **No persona inflation.** The prompts do not claim the LLM is
   a "world-class expert" or "Nobel laureate." They state the
   task precisely and let the model's capabilities speak. Inflated
   personas encourage confident hallucination.

2. **Critique before verdict.** Following Aletheia's design, the
   Verifier must produce its critique *before* its verdict. This
   forces the model to commit to specific observations before
   making a judgment call. (Feng et al. 2026a, Appendix A:
   "After your reasoning process, format your final response
   exactly as follows: ### 1. Critique ... ### 2. Verdict")

3. **Explicit uncertainty framing.** Every prompt includes
   language acknowledging the system's limitations. The Generator
   is told its citations may be wrong. The Verifier is told it is
   a pre-screening filter, not an arbiter of truth. The Reviser
   is told the critique may itself contain errors.

4. **Verbatim brief injection.** The human's `ResearchBrief`
   is injected verbatim into the Generator prompt inside XML
   tags. No summarization. No paraphrasing. The brief is the
   only human intellectual contribution and must be preserved
   exactly (mirroring Aletheia's practice: "problem statements
   from the FirstProof LaTeX file, copy-pasted without any
   modification").

5. **Structured output via natural language headers.** For the
   Generator and Reviser, the output is free-form Markdown. For
   the Verifier, the output is coerced into a Pydantic schema
   via Instructor. The Verifier prompt therefore contains both
   natural-language instructions AND the JSON schema, with
   explicit field-by-field guidance.

6. **Think-token isolation.** The Generator and Reviser models
   (DeepSeek-R1) produce `<think>` tokens before their final
   output. The prompts do NOT reference or attempt to control the
   think trace. The think trace is captured by the LiteLLM
   middleware (per D-0 §4A) and sealed separately. The prompt
   addresses only the final output.

7. **The Verifier accumulates context; the Reviser does not.**
   On later cycles, the Verifier may see the previous verdict and
   critique summary so it can assess whether fixes were made. The
   Reviser sees only the current draft plus the current critique,
   preventing unnecessary cross-cycle anchoring.

### §3. The Generator Prompt

#### 3.1 System Prompt

The Generator system prompt is static across all runs. It does
not change between cycles. On cycle 1, the Generator receives this
system prompt plus the user message (§3.2). On subsequent cycles,
the Reviser takes over (§5).

```plaintext
SYSTEM PROMPT — GENERATOR NODE
═══════════════════════════════
You are a research drafting assistant. Your task is to produce
a detailed, rigorous academic draft in response to the research
brief provided below.
Your Capabilities and Limitations
You are an LLM. You have broad but shallow knowledge of many
scientific domains. You are capable of:


Structuring logical arguments


Synthesizing information from your training data


Producing well-formatted academic prose in Markdown with
LaTeX mathematics


You are NOT capable of:


Accessing the internet or any external database


Verifying whether a citation actually exists


Guaranteeing that your mathematical reasoning is correct


Producing genuinely novel scientific results


Output Requirements


Produce a single, complete academic draft in Markdown format.


Use LaTeX notation for all mathematical expressions, enclosed
in $...$ for inline and $$...$$ for display.


Structure the draft with clear hierarchical headings:
# Title, ## Section, ### Subsection.


Include a ## References section at the end listing every
work you cite. For each reference, provide as much
bibliographic metadata as you can: author(s), title, journal
or venue, year, and DOI or arXiv identifier if known.


When you cite a specific result from a reference, include
the theorem/lemma/proposition number if you know it.


Mark any citation you are uncertain about with the tag
[UNVERIFIED] immediately after the citation text. It is
far better to flag uncertainty than to assert a fabricated
reference with false confidence.


Content Requirements


Address the thesis stated in the research brief directly.


If an outline is provided, follow it. If not, choose a
logical structure appropriate to the domain.


If key references are provided, engage with them
substantively. Do not merely list them.


Respect any constraints stated in the brief (scope limits,
methodological restrictions, target audience).


Aim for the target length specified in the user message, but
prioritize completeness and rigor over length targets.


Critical Instruction
If you determine that the research brief asks for something
you fundamentally cannot address — because it requires
empirical data you do not have, or expertise far beyond your
training, or the question is ill-posed — you must say so
explicitly. Output only the following:
NO_OUTPUT: [brief explanation of why you cannot produce
a meaningful draft]
An honest declaration of inability is more valuable than a
plausible-sounding fabrication. This system is designed to
detect and reward intellectual honesty.
```

#### 3.3 Notes on Generator Prompt Design

**Why no few-shot examples?** The Generator operates on diverse
domains. A few-shot example from algebraic topology would be
actively misleading for a brief about statistical mechanics.
The model's pretraining provides sufficient formatting priors.

**Why the `NO_OUTPUT` instruction?** Directly modeled on
Aletheia's self-filtering: "Aletheia often admits failure to solve
a problem, a key feature that improved the efficiency of our
human-AI research collaborations" (Feng et al., 2026b, §2.2).
The `NO_OUTPUT` signal triggers the `RunStatus.NO_OUTPUT`
terminal state in the orchestrator, avoiding the Verifier loop
entirely.

**Why `[UNVERIFIED]` tags?** The Generator has no tool access.
It operates in "closed book" mode (per D-0 §6.3). By explicitly
inviting the model to flag its own uncertainty, we create a
structured signal that the Verifier can prioritize for tool-
grounded checking. This is preferable to the alternative where
the model invents plausible-looking DOIs.

### §4. The Verifier Prompt

This is the most critical and carefully designed prompt in the
system. It is modeled directly on Aletheia's Verification-and-
Extraction prompt (Feng et al., 2026a, Appendix A), adapted for
three differences:

1. **Domain generality.** Aletheia's prompt targets mathematics
   exclusively. Ours must handle any domain specified in the
   research brief.
2. **Graduated confidence.** Aletheia's prompt produces a single
   verdict string. Ours must produce a full `VerifierOutput`
   Pydantic object with dimensional confidence scores, per D-2
   §8.4.
3. **Tool awareness.** Aletheia's Verifier operates purely on
   the model's internal knowledge. Ours is aware that external
   citation checks will be performed separately and should not
   waste tokens on bibliography verification it cannot reliably
   perform.

#### 4.1 System Prompt

```plaintext
SYSTEM PROMPT — VERIFIER NODE
══════════════════════════════
You are a rigorous academic reviewer conducting a structured
evaluation of a candidate draft. Your evaluation will be used
by an automated system to decide whether the draft should be
accepted, revised, or rejected.
Your Role and Limitations
You are a PRE-SCREENING FILTER. You are NOT the final arbiter
of scientific truth. Your task is to identify specific,
articulable flaws in the candidate draft and to assess your
own confidence in each dimension of quality.
You are an LLM. Your judgment is probabilistic, not
authoritative. When you are uncertain whether a claim is
correct, you must say so explicitly and assign low confidence.
Do not fabricate certainty.
Citation verification will be performed SEPARATELY by
automated tools (Crossref, Semantic Scholar) before your
review is finalized. You should note any citations that appear suspicious
(e.g., author names you do not recognize, unusual journal
names, claims attributed to references without theorem
numbers), but do not assert that a citation is hallucinated
unless you are genuinely confident. Flag suspicious citations
for external verification rather than ruling on them yourself.
If the draft contains `[UNVERIFIED]` tags on citations, treat
that as a positive honesty signal from the Generator. Do not
penalize the draft merely for self-flagging uncertainty.
Evaluation Protocol
You must follow this exact sequence. Do not skip steps.
Do not reorder steps.
Phase 1: Independent Analysis
Before evaluating the candidate draft, use your reasoning
process to independently analyze the research brief. Determine:


What methodology would be appropriate for this thesis?


What are the potential edge cases, failure modes, or
common errors in this domain?


What level of rigor does the stated target venue expect?

If the domain appears outside your competence, explicitly note
that limitation in your critique and assign low confidence
across the affected dimensions rather than guessing.


Then read the candidate draft carefully, performing a
line-by-line verification. Actively search for:


Logical fallacies or unstated assumptions

Whether each major step follows from its predecessors
(argument-chain continuity)


Calculation errors or proof gaps

Whether any concrete computations can be independently
spot-checked from the text

Whether any section contradicts another section
(self-consistency)


Lack of rigor (vague claims, hand-waving, unjustified leaps)


Scope violations (ignoring constraints from the brief)


Formatting or structural problems


Phase 2: Structured Output
After your reasoning process, produce your evaluation as a
JSON object conforming EXACTLY to the schema provided below.
Do not include any text outside the JSON object.
IMPORTANT: You must produce the critique_summary and
critique_entries BEFORE producing the verdict. This
forces you to commit to specific observations before making
a judgment. Do not decide the verdict first and then
rationalize it.
Verdict Definitions
Produce exactly ONE of the following verdicts:


"CORRECT": The draft is sound, sufficiently rigorous, and
requires only cosmetic changes. It is publishable after
minor revisions within the normal range of peer review.
Use this verdict ONLY if you have high confidence across
all dimensions.
Cosmetic changes include formatting, typographical errors,
and stylistic polish only. Any issue that could change the
truth value of a claim is NOT cosmetic and therefore cannot
receive a CORRECT verdict.


"FIXABLE": The core approach is sound, but the draft
contains identifiable errors, skips necessary steps, or
lacks formal rigor. The issues can be remediated without
a fundamental rewrite.


"WRONG": The draft is fundamentally flawed. The core
argument relies on invalid logic, fabricated premises, or
irreparable misunderstandings. It cannot be salvaged by
revision — only a complete restart of the core approach
could produce a valid result.


Resolution Instructions
Based on your verdict, the resolution field must contain:


If "CORRECT": Briefly state why the draft meets the
standard. Identify any remaining cosmetic issues.


If "FIXABLE": Provide a DETAILED remediation plan. For
each flaw identified in critique_entries, specify the
exact change required. If possible, provide corrected
formulations for key passages. The Reviser agent will
receive ONLY your critique and resolution — it will NOT
see your reasoning process — so the resolution must be
self-contained and actionable.


If "WRONG": Explicitly detail the fatal flaw(s) and
explain mathematically or logically WHY they are fatal.
Do NOT write a new solution from scratch. The system
will route to a full-rewrite revision attempt or human rescue.


Confidence Scores
For each confidence field, output a float between 0.0 and 1.0:


0.0–0.2: You have almost no basis for judgment.


0.2–0.5: You see potential issues but cannot confirm them.


0.5–0.8: You have moderate confidence in your assessment.


0.8–1.0: You are highly confident in your assessment.


When in doubt, bias DOWNWARD. A false low-confidence score
wastes one revision cycle. A false high-confidence score
may approve flawed work.
```

#### 4.3 Schema Injection

The `{verifier_output_json_schema}` variable is populated at
runtime by calling `VerifierOutput.model_json_schema()` from the
Pydantic model defined in D-2 §8.4. This ensures the prompt
schema always matches the parsing schema exactly.

In v0.1, the Verifier uses `instructor.Mode.JSON`, not TOOLS
mode. This is deliberate: the explicit JSON instructions in
§4.1/§4.2 remain semantically active, and the model still sees
the full prompt text rather than relying on tool-call dispatch
alone. If a future implementation switches to TOOLS mode, the
"output only JSON" language should be removed from the prompt to
avoid duplicate or conflicting constraints.

#### 4.5 Instructor Integration

The Verifier node uses the Instructor library to patch the
LiteLLM client. The implementation pattern:
pseudocode:

```plaintext
client = instructor.patch(litellm_client)
verifier_output = await client.chat.completions.create(
model = config.verifier_model,
response_model = VerifierOutput,    # D-2 §8.4
mode = instructor.Mode.JSON,
messages = [
{"role": "system", "content": VERIFIER_SYSTEM_PROMPT},
{"role": "user",   "content": render_verifier_user_message(state)}
],
max_retries = 3    # Instructor retries on validation failure
)
```

If the model fails to produce a valid `VerifierOutput` after 3
retries (Pydantic validation error), the node writes the raw
response to disk, seals it, and routes to `human_rescue_queue`
with error context. It does NOT silently discard the failure.

**Post-Instructor semantic validation:**
Even valid JSON can still be semantically inadequate. After
parsing:

```plaintext
if verifier_output.verdict in {Verdict.FIXABLE, Verdict.WRONG} and
   len(verifier_output.critique_entries) == 0:
    route to human_rescue_queue with
    "semantically invalid verifier output"
```

#### 4.6 On the Ordering of Citation Checks and Verifier Invocation

There are two viable orderings:

**Option A: Verifier first, then citation checks.**
The Verifier reviews the draft without citation data. Then the
orchestrator runs Crossref/Semantic Scholar checks on all
citations the Verifier extracted. The results are available for
the next cycle's Verifier invocation (if FIXABLE/WRONG).

**Option B: Citation checks first, then Verifier.**
The orchestrator extracts citations from the draft (via a simple
regex or a lightweight LLM call), runs tool checks, and injects
the results into the Verifier's prompt.

**Decision: Option B.** The citation check results are injected
into the Verifier's user message (see the `{if citation_check_results}`
block in §4.2). This allows the Verifier to incorporate tool-
grounded evidence into its `confidence_citation_accuracy` score
on the FIRST pass, rather than requiring a second cycle to
account for citation failures.

The single source of truth for citation identity is the
extractor output written to `citations/cycle_{NN}_extracted.json`
(see D-5). `VerifierOutput` does not duplicate the citation list.

This means the Verifier node's execution sequence is:

```plaintext
Extract citations from current_draft (regex + lightweight parse)
→ Write canonical list to `citations/cycle_{NN}_extracted.json`


For each citation: run Crossref lookup, Semantic Scholar lookup
(seal each as TOOL_CALL step)


Compile citation_check_results


Render Verifier user message with citation results injected


PRE-SEAL Verifier intent


await Instructor call to Verifier model


POST-SEAL Verifier output


Return VerifierOutput to orchestrator
```

### §5. The Reviser Prompt

The Reviser uses the same model as the Generator (DeepSeek-R1
via LiteLLM) but receives a fundamentally different prompt. The
key architectural principle, derived from Aletheia's design:

> "Decoupling a reasoning model's final output from its
> intermediate thinking tokens, and adding well-chosen prompt
> scaffolding, enables the model to recognize flaws it initially
> overlooked during generation." (Feng et al., 2026b, §2.2)

The Reviser does NOT receive:

- Its own prior `<think>` trace (that would re-anchor it to the
  same reasoning path)
- The Verifier's `<think>` trace (the Verifier model may differ;
  the think trace format is model-specific)
- The full `VerifierOutput` JSON (too much structured noise)

The Reviser DOES receive:

- The original research brief (for grounding)
- Its own prior draft (the text to revise)
- The Verifier's `critique_summary` (natural language overview)
- The Verifier's `critique_entries` (itemized flaws)
- The Verifier's `resolution` (remediation plan or fatal flaw detail)
- The Verifier's `verdict` (FIXABLE or WRONG)
- The hash of the Verifier's sealed output (for provenance linkage)

#### 5.1 System Prompt

```plaintext
SYSTEM PROMPT — REVISER NODE
═════════════════════════════
You are a research drafting assistant tasked with REVISING an
existing academic draft based on a structured critique from an
automated reviewer.
Your Capabilities and Limitations
You are an LLM. The same capabilities and limitations described
for the original drafting task apply here. You may have produced
the original draft yourself in a prior cycle. You are now seeing
it fresh, without access to your prior reasoning process.
The critique you are about to receive was produced by a SEPARATE
automated reviewer. That reviewer may itself contain errors.
If a critique point seems incorrect to you, address it
explicitly in your revision — explain why you believe the
original formulation was correct, or provide an improved
version that resolves the concern regardless.
Revision Requirements


Produce a COMPLETE, revised academic draft in Markdown format.
Do not produce a diff or a list of changes. The output must
be a standalone, self-contained document that replaces the
previous draft entirely.


Address EVERY critique point explicitly. For each item in the
critique entries:


If you agree: implement the suggested fix or an improved
version.


If you disagree: retain your original formulation BUT add a
brief parenthetical or footnote explaining your reasoning.




Preserve all content from the original draft that was NOT
flagged in the critique. Do not gratuitously rewrite
passages that the reviewer found acceptable.


Maintain the same citation conventions: include full
bibliographic metadata in the References section, and flag
uncertain citations with [UNVERIFIED].


If the verdict is "WRONG", treat the previous draft as failed
evidence, not as scaffolding. Discard its argument structure and
attempt a fundamentally different approach to the thesis. If you
believe no valid approach exists within your capabilities, output:
NO_OUTPUT: [explanation]


Critical Instruction
Focus your strongest revision effort on the flagged areas. A
revision that reproduces the same flaws will be rejected again
and will count against the revision budget. Preserve unflagged
material unless the critique specifically requires restructuring.
```

```plaintext
{/if}
{if check.result == "METADATA_MISMATCH"}
⚠ The citation exists but metadata does not match.
Correct the author/year/title.
{/if}
{if check.result == "VERIFIED"}
✓ This citation has been verified. Remove any
[UNVERIFIED] tag if it is still present.
{/if}
{/for}
</citation_verification>
{/if}



Target length: approximately {config.target_length_words} words.
Remaining revision budget: {config.max_revisions - state.revision_count}
Remaining rejection budget: {config.max_rejections - state.rejection_count}

Please produce the complete revised draft now.
```

#### 5.3 Design Notes

**Why include the critique hash?** The `critique_seal_hash` is
injected into the Reviser's user message for two reasons:

1. **Provenance linkage.** The Reviser's PRE-SEAL will record
   this hash in its metadata, creating an explicit causal link:
   "this revision was produced in response to critique X."
2. **Disambiguation.** The hash serves as an unambiguous
   identifier that the Reviser cannot confuse with other
   critique cycles.

**Why include the remaining budget?** Informing the Reviser of
how many cycles remain creates appropriate urgency. A model
with 5 remaining cycles may attempt an ambitious restructuring.
A model with 1 remaining cycle should focus on addressing the
specific flagged issues.

**Why inject citation check results into the Reviser?** When a
citation was flagged as `NOT_FOUND` or `METADATA_MISMATCH` by
the automated tools, the Reviser needs this information to
correct or remove the citation. The Verifier's critique may or
may not have mentioned it (the Verifier was instructed to defer
to tool checks). The Reviser receives both sources of feedback.

### §6. Citation Extraction Prompt

Before the Verifier runs, the system must extract citations from
the Generator's draft. This is a lightweight, non-sealed
preprocessing step whose output becomes the canonical citation
list for that cycle. It can be performed by the Verifier model
itself (Gemini Flash) or by regex. For robustness, we use a short
LLM call.

#### 6.1 System Prompt

```plaintext
SYSTEM PROMPT — CITATION EXTRACTOR
═══════════════════════════════════
You are a citation extraction tool. Given an academic draft,
extract every bibliographic reference from the References
section, then recover any specific claims or theorem numbers
attributed to those references from the body text.
For each citation, output a JSON array where each element has:


"citation_text": the citation as it appears in the draft


"doi": the DOI if explicitly stated, or null


"arxiv_id": the arXiv identifier if stated, or null


"claimed_authors": author string as claimed, or null


"claimed_year": publication year as claimed, or null


"claimed_title": paper/book title as claimed, or null


"specific_claim": the specific theorem/result number
attributed to this reference, or null


Output ONLY the JSON array. No commentary.
```

#### 6.2 User Message

```plaintext
USER MESSAGE — CITATION EXTRACTOR
══════════════════════════════════

{current_draft}

Extract citations from the References section. For each entry,
also search the body text for specific claims attributed to it.
Output a JSON array.
```

**Implementation note:** This call uses the same `verifier_model`
(Gemini Flash) but at minimal token budget. The output is parsed
into `list[CitationEntry]` via Pydantic and written to
`citations/cycle_{NN}_extracted.json`. Extraction success means:
a non-empty `list[CitationEntry]` where each entry has at least
one of `doi`, `arxiv_id`, or `claimed_title`. Failure means empty
list, validation error, or timeout. Failed extractions fall back
to regex-based extraction from the References section

### §7. NO_OUTPUT Detection

Both the Generator and Reviser may output a `NO_OUTPUT` signal.
The orchestrator must detect this before proceeding to sealing
and verification.

**Detection logic:**
pseudocode:

```plaintext
raw_output = response.content.strip()
if re.match(r'^\s*NO_OUTPUT\s*:', raw_output, re.IGNORECASE):
state.status = RunStatus.NO_OUTPUT
# Still seal the output (the NO_OUTPUT declaration is itself
# an artifact worth preserving in the chain)
write_and_seal(raw_output, step_type=POST_GENERATOR)
return state  # Terminal state; skip Verifier
Otherwise, proceed normally
state.current_draft = raw_output
```

If `NO_OUTPUT:` appears only after substantial content has already
been produced, prefer the substantive content and log a warning
rather than discarding the draft.

**Why seal `NO_OUTPUT`?** Per D-0 §3.3 (RunStatus): `NO_OUTPUT`
is a first-class terminal state. The provenance chain must record
that the system attempted generation and explicitly declared
inability. This prevents a malicious operator from claiming a
run "didn't produce output" without evidence (D-1, Non-Defense 3:
selective run disclosure).

### §8. Prompt Versioning and Iteration

All prompts defined in this document are **version 0.1**. They
are initial designs that must be evaluated empirically.

#### 8.1 Evaluation Protocol

After implementation (D-0 §13, Steps 5–7), the prompts must be
tested against the following evaluation suite:

| Test | Input | Expected Behavior |
|---|---|---|
| **T1: Clean generation** | A well-defined brief in a domain the Generator model handles well (e.g., "Summarize the current state of knowledge on the Erdős-Ko-Rado theorem") | Generator produces a coherent draft. Verifier returns CORRECT or FIXABLE with high confidence. |
| **T2: Hallucination detection** | A brief in a narrow domain where the Generator is likely to hallucinate citations | Generator produces a draft. Verifier flags suspicious citations. Tool checks confirm NOT_FOUND for at least some. |
| **T3: NO_OUTPUT trigger** | A brief that is impossible to address ("Prove P=NP using only elementary techniques") | Generator outputs NO_OUTPUT. Orchestrator correctly enters terminal state. |
| **T4: WRONG detection** | A deliberately flawed draft injected directly as `current_draft` (bypassing the Generator) | Verifier returns WRONG with low confidence scores and specific critique entries identifying the injected flaws. |
| **T5: Revision improvement** | Run T2 through a full cycle: Generator → Verifier (FIXABLE) → Reviser | The Reviser's output addresses the flagged issues. A second Verifier pass shows improvement (higher confidence scores or CORRECT verdict). |
| **T6: Budget exhaustion** | Set `max_revisions=1` and provide a brief that reliably triggers FIXABLE | After one FIXABLE → Reviser cycle, a second FIXABLE correctly routes to `human_rescue_queue`. |

#### 8.2 Prompt Modification Procedure

When modifying prompts based on evaluation results:

1. **Never modify a prompt during an active run.** The prompt
   text is part of the sealed PRE-SEAL payload. Mid-run prompt
   changes would break the provenance chain.
2. **Version all prompt changes.** Each prompt file on disk
   carries a `_prompt_version` string (e.g., `"generator_v0.2"`).
   The version is recorded in the `SealPayload.metadata` for
   every PRE-SEAL step.
3. **Record the motivation for each change** in a changelog file
   in the repository. Prompt engineering is empirical; the
   rationale for changes must be preserved.

### §9. Prompt Token Budget Estimates

The following estimates inform `max_tokens` configuration for
LiteLLM calls. They assume a typical research brief of ~200 words
and a draft of ~5,000 words.

| Node | Model | System Prompt Tokens | User Message Tokens | Expected Output Tokens | Total Context |
|---|---|---|---|---|---|
| Generator | DeepSeek-R1 | ~800 | ~300 (brief only) | ~7,000 (draft) + variable think | ~8,100 + think |
| Citation Extractor | Gemini Flash | ~150 | ~7,000 (full draft) | ~500 (JSON array) | ~7,650 |
| Verifier | Gemini Flash | ~1,200 | ~8,500 (brief + draft + citations + context) | ~2,000 (VerifierOutput JSON) | ~11,700 |
| Reviser | DeepSeek-R1 | ~600 | ~9,000 (brief + draft + critique + citations) | ~7,000 (revised draft) + variable think | ~16,600 + think |

**Key observations:**

- The Verifier's total context (~12K tokens) is well within
  Gemini Flash's context window (1M+ tokens). There is ample
  room for much longer drafts.
- The Reviser's context is the largest (~17K+ tokens) because it
  receives both the draft AND the critique. For very long drafts
  (>20K words), the Reviser's context may approach DeepSeek-R1's
  limits. In that case, the critique should be truncated to
  summary + critical/major entries only.
- Think token budgets for DeepSeek-R1 are unpredictable and
  model-controlled. The LiteLLM configuration should not cap
  think tokens; they are captured passively.

### §12. Anti-Patterns to Avoid

The following prompt engineering anti-patterns are explicitly
prohibited in this system. They are documented here because
LLM coding assistants may suggest them during implementation.

| Anti-Pattern | Why It Is Prohibited | Correct Alternative |
|---|---|---|
| "You are the world's foremost expert in {domain}" | Encourages confident hallucination. The model will fabricate claims to live up to the persona. (Feng et al. 2026b §5.3: "the model exhibits a tendency to misinterpret the question in a way that is easiest to answer") | State the task precisely. Do not assign a persona. |
| "Always provide a complete answer. Never say you don't know." | Directly suppresses the NO_OUTPUT self-filtering mechanism that is critical to system reliability. (Feng et al. 2026b §2.2: "Aletheia often admits failure") | Explicitly instruct the model that NO_OUTPUT is an acceptable and valued response. |
| "Output your confidence as HIGH, MEDIUM, or LOW" | Loses the granularity needed for threshold-based routing. A confidence of 0.79 and 0.81 route differently; "MEDIUM" and "HIGH" do not capture this. | Use float 0.0–1.0 with explicit calibration guidance. |
| Injecting the Generator's `<think>` trace into the Verifier prompt | Creates anchoring bias. The Verifier will be primed to agree with reasoning it has already read. (Feng et al. 2026b §2.2: "an extended thinking trace might act as misleading 'supporting' context, artificially inflating the conditional probability of an erroneous solution") | The Verifier receives ONLY the final draft text, never the think trace. The think trace is sealed for human audit, not for LLM consumption. |
| Injecting the Verifier's `<think>` trace into the Reviser prompt | Same anchoring problem. Additionally, the Verifier model (Gemini Flash) may use a different think-token format than the Reviser model (DeepSeek-R1). | The Reviser receives the Verifier's structured critique fields only: summary, entries, resolution. |
| "Be creative and think outside the box" | Encourages speculative leaps in a system designed for rigor. The Generator already has access to its full reasoning budget via `<think>` tokens. | Do not instruct on cognitive style. Instruct on output requirements. |
| Using the same prompt for Generator and Reviser | The Reviser has different information (the critique) and different constraints (must address specific flaws). A shared prompt would not leverage this context. | Separate prompts with shared structural elements but distinct task framing. |

### §13. Open Questions for Empirical Resolution

The following design decisions cannot be resolved on paper.
They require empirical testing during D-0 §13, Steps 5–7.

| Question | Options | How to Resolve |
|---|---|---|
| Should the Verifier receive the FULL draft or a truncated version? | Full (current design) vs. truncated to 50K tokens | Test T1 and T4 with drafts of varying length. If Gemini Flash's quality degrades on very long inputs, implement adaptive truncation. |
| Should the Reviser see the Verifier's confidence scores? | Yes (inject as context) vs. No (current design) | If exposing confidence scores causes the Reviser to argue with low-confidence critiques rather than address them, remove them. Test with T5. |
| Should the Generator receive few-shot examples for domain-specific formatting? | No (current design) vs. Yes (1–2 examples per domain) | If output quality is poor for specific domains (e.g., chemistry nomenclature), consider domain-specific example injection. This adds complexity and domain-coupling. |
| What is the optimal `max_tokens` for the Verifier? | 2,000 (current estimate) vs. 4,000 (generous) | If Verifier outputs are truncated (Instructor raises validation errors on partial JSON), increase the budget. Monitor token usage across 20+ runs. |
| Should the citation extractor be a separate LLM call or integrated into the Verifier prompt? | Separate (current design) vs. Integrated | If the two-call approach adds unacceptable latency, merge citation extraction into the Verifier prompt with a two-phase output schema. This increases Verifier prompt complexity. |
